1.2 考前 30 分钟上场卡：别重学，叫醒动作

这块是考试当天早上或进考场前最后一轮看的。你只做三件事：认题眼、说第一步、避开最常见扣分点。如果这张卡看完还想重读整章，先忍住，考前最后半小时重读长讲义的收益很低。

30 分钟分配：SQL/关系代数 8 分钟，事务/恢复 7 分钟，范式 7 分钟，ER 5 分钟，最后 3 分钟只看你自己的 6 句错题口令。
卷面长相第一动作标准第一句话最容易扣分
SQL：最高、全部、没有、每组
题干出现每个部门、所有课程、没有选课、人数超过 10。		 先写草稿：FROM/JOIN -> WHERE -> GROUP BY -> HAVING -> SELECT。 “本题先确定来源表和连接条件，再判断条件是筛行还是筛组。” 把 COUNT(*) > 10 写进 WHERE；每组最高不回表；“全部”只写普通 JOIN。
关系代数：σ、Π、×、⋈、−、÷
题目明说“用关系代数表达”。		 先圈输出列，最外层通常是 Π；再圈筛选条件 σ；跨表用连接或 × + σ。 “我先把中文查询翻成输出、条件、来源表，再选择关系代数符号。” 只写 SQL；把筛行写成投影；看到“所有/全部”忘记除法或差集思路。
事务调度：r/w/lock/unlock
题干出现 T1、T2、r1(A)、w2(A)、lock、unlock。		 先分题型：读写调度画冲突图；锁调度找第一次 unlock；严格 2PL 看写锁是否保留到 commit/abort。 “冲突只看不同事务、同一数据项、至少一个写；谁先发生边就从谁指向谁。” 读读也画边；不同数据项乱连边；把 commit 顺序当成冲突串行顺序。
恢复日志：commit/crash/old/new
题干给 START、COMMIT、crash、old value/new value。		 先分 winner 已提交 和 loser 未提交，再决定 Undo/Redo。 “恢复题第一步不是算数值，而是先标出崩溃时哪些事务已提交、哪些未提交。” 撤销已提交事务；保留未提交事务新值；Undo/Redo 方向写反。
范式：R(...) 和 F={...}
题干问候选码、BCNF、无损、保持依赖。		 先看右部没出现的属性，再求闭包；候选键出来后才判 3NF/BCNF。 “我先用属性闭包求候选键，因为范式判断必须知道哪些左部是超码。” 不求闭包直接猜键；BCNF 用了 3NF 的主属性例外；无损和保持依赖混为一谈。
ER：自然语言业务需求
题干给诊所、银行、学生选课、预约、处方等场景。		 先圈实体和联系，再标基数；M:N 或带属性的联系单独成表。 “我先抽实体、属性、联系和基数，再把 ER 图转换成关系模式。” 只列实体不画联系；漏基数；把成绩、处方明细这类联系属性硬塞到一端实体。
遇到不会的大题

先写题型和第一步，不要空白。例如“这是恢复日志题，先分已提交/未提交事务”。哪怕后面没算完，阅卷至少看到你会入门。

遇到选择题

先说边界：WHERE/HAVING、ACID/CAP、GRANT/GRANT OPTION、触发器/存储过程、外键父子方向。能解释错项，才算会。

遇到 ER 图

不要画漂亮但空的图。实体名、主键、关键属性、联系名、1/N/M/N、联系实体化，比装饰重要得多。

1.5 客观题排雷：2023A 和 2020 都不是纯大题卷

数据库最后一天不能只练 SQL 大题。2023A 参考卷开头就是多项选择，2020 卷也有单选和判断。客观题通常不让你长篇计算，而是把相邻概念、方向、触发条件放在一起考。下面这些是考前最值得见一眼的卷面形态。

做法：每题先说“它在考哪条边界”。如果你能说出为什么错项错，选择题才算真的会。
关系代数等价

卷面：R ⋈_{R.B<S.C} S 等价于什么？

答案：σ_{R.B<S.C}(R × S)。带条件的连接就是先做笛卡尔积，再按条件选择。

错项诱因：把条件写到投影 Π 上，或者忘记先有 R × S。

联合索引最左前缀

卷面：R(A,B) 上有联合索引，哪些查询能用？

口诀：条件里有最左字段 A，通常能用；只写 B，通常用不上；对 A 做函数/表达式也可能失效。

错项诱因：以为索引建在 (A,B) 上，就单独查 B 也一样快。

ACID / CAP / BASE

卷面：事务处理应该遵循什么原则？NoSQL 又常提什么一致性模型？

事务：ACID。分布式/NoSQL：CAP 是不可能三角，BASE 是基本可用、软状态、最终一致。

错项诱因：把 CAP 当成“事务原则”。

授权传播

卷面：谁可以把表权限再授予别人？

答案：拥有该权限并带 WITH GRANT OPTION 的用户。

错项诱因：以为“有 SELECT 权限”就一定能授权别人。普通使用权不等于传播权。

触发器

卷面：触发器是不是用户手动调用？能不能有 SELECT 触发器？行级一定更快吗？

判断：触发器满足事件自动触发；常见事件是 INSERT/DELETE/UPDATE；行级对每行触发，语句级对每条语句触发。

错项诱因：把触发器当普通存储过程，或者以为触发次数越多越快。

外键级联方向

卷面：ON DELETE SET NULL / CASCADE 判断对错。

口诀：父表被删时，子表怎么办。删除子表一行不会反过来删除父表。

错项诱因：把引用方向想反。

客观题信号先想什么回炉入口
联合索引 / 最左前缀 / 函数索引索引能不能用，不只看有没有索引，还看 WHERE 是否命中最左字段、是否对列做函数。本页客观题排雷 + Day5 题眼速查
GRANT / WITH GRANT OPTION / REVOKE区分“能用权限”和“能传播权限”。本页客观题排雷
触发器 BEFORE/AFTER/INSTEAD OF/FOR EACH ROW触发时机、触发事件、触发粒度三件事分开看。本页客观题排雷
完整性 / 实体完整性 / 参照完整性 / 用户自定义完整性实体完整性看主键；参照完整性看外键；用户自定义常靠 CHECK 或触发器。Day5 外键题 + 本页客观题排雷
视图 / WITH CHECK OPTION视图是外模式/虚表；CHECK OPTION 限制通过视图更新后的元组仍满足视图条件。主页真题桥接“视图”卡

2. 往年题题眼矩阵：同一类题会换皮出现

你现在不需要一口气刷完整张旧卷。最后一天真正值钱的是：看到换了表名、换了故事背景的题，还能立刻认出它属于哪一类。下面这张表把 SQL1/SQL2、2020、2023A 和事务恢复小卷里的高频信号压成题眼 -> 模板动作 -> 下一步训练。

用法：这不是让你机械背旧题答案，而是让你知道“同一题型换皮后仍然调用同一套动作”。比如“选修所有课程”“所有学生都选修”“参加全部剧目”中文故事不同，本质都是全称查询。
来源真题信号模板动作下一步训练
SQL1 / SQL2“每门课平均成绩”“人数超过阈值才统计”“按人数降序/课程号升序”GROUP BY + HAVING + ORDER BY。先分组，再筛组，最后排序。练 3 句口令：WHERE 筛行；HAVING 筛组；ORDER BY 不改变分组结果，只改变显示顺序。
SQL1 / SQL2“年龄大于所有女学生年龄”“年龄大于女学生平均年龄”“所有”用 > ALL 或大于最大值；“平均”用 AVG。两者不是一回事。看到“所有”先想最大值/ALL；看到“平均”才想 AVG，别把高于平均误写成高于所有人。
SQL1 / SQL2“选修了所有课程”“全部学生都选修的课程”SQL 用双重 NOT EXISTS 或 COUNT(DISTINCT)=总数；关系代数用除法。练“反过来想”：不存在一个课程/学生被漏掉。
SQL1 / SQL2 / 2020插入女生物理选课、删除选修某老师课程记录、更新低于平均成绩DML + 子查询：INSERT INTO ... SELECT、DELETE WHERE ... IN/EXISTS、UPDATE ... WHERE。先写要改哪张表，再写筛选条件从哪些表查出来，避免把 SELECT 写成孤立查询。
2020“演过 Great Writer 的某些剧，但从未演过 Nobody 的剧；不要重复”EXISTS + NOT EXISTS + DISTINCT。先保留存在，再排除存在。把“some/never/no duplicate”翻译成“存在/不存在/去重”。
2020“列出所有演员和参加过的剧，包括没有 Role 记录的演员”LEFT JOIN。保留左边全体，再接可能存在的右边记录。看到“包括没有记录的人/物”，不要从中间表出发。
Transaction / 2020 / 2023Aread/write 调度、lock/unlock 序列、old/new log、commit/crash先分流：read/write 画冲突图；lock/unlock 查 2PL；log/crash 分 winner/loser。不要把事务题混成一团。每种题的第一动作不同。
2020 / 2023A外键 ON DELETE SET NULL / CASCADE；插入不存在父元组先看父表和子表方向。父表删除触发对子表动作；子表插入必须找到父表被引用元组。把“谁引用谁”画成箭头，防止级联方向写反。
2020 / 2023A / 范式笔记R(A,B,C,D,E), F={...}，求候选键、BCNF、保持依赖、无损连接右部没出现的属性先带上；求闭包；判断左部是否超码；分解后分别查保持依赖和无损。严格按“闭包 -> 候选键 -> 违反依赖 -> 分解 -> 两个性质”写，不跳步。
2020 / 2023A诊所、银行、演员剧目等业务需求，要求 ER 图和关系模式圈实体、属性、联系、基数；M:N 或带属性联系单独成表；不确定处写假设。先抽名词/动词，不要上来画图。画完再转关系模式。

4. 六个排雷口令：考前必须能脱口而出

SQL 顺序FROM/JOIN -> WHERE -> GROUP BY -> HAVING -> SELECT -> ORDER BY。WHERE 筛行，HAVING 筛组。
全部/所有SQL 用双重 NOT EXISTS 或计数法；关系代数优先想到除法。
最高/最多先求 max/avg/count，再回表匹配；并列最高要都保留。
候选键右边从未出现的属性通常必须带；闭包推全体，再查最小性。
事务read/write 画冲突图；lock/unlock 查第一次 unlock；log/crash 先分已提交和未提交。
ER名词多为实体/属性，动词多为联系；M:N 和带属性联系要单独成表。
最危险错法：看到“数据库题”就直接写 SQL。卷子里 SQL、关系代数、范式、事务、恢复、ER 是不同语言。先判题型，再写答案。
关系代数符号最后一眼：σ 筛行，Π 筛列，× 先暴力组合，⋈ 按条件拼表，− 表示“没有/排除”，÷ 表示“全部/所有”，ρ 给同一张表起别名。卷面如果要求“用关系代数表示”，答案里就应该出现这些符号，而不是直接写 SQL。

6. 半套模拟：先写题眼和骨架，再展开解析

下面 6 题不是为了刷量，是为了覆盖 2023A 和往届题反复出现的硬分区。每题先写 3 行：题眼、第一步、最怕错点。

题 1｜SQL：各部门最高工资员工，保留并列

emp(num, name, dept, salary, boss)。写 SQL：查询每个部门工资最高的员工编号、姓名、部门和工资。如果并列最高，全部输出。

展开答案与解析
题眼“每个部门最高”不是简单 MAX，而是分组聚合后回表。题目要求并列都输出，所以不能只排序取一条。
SELECT e.num, e.name, e.dept, e.salary
FROM emp e
JOIN (
  SELECT dept, MAX(salary) AS max_salary
  FROM emp
  GROUP BY dept
) m
ON e.dept = m.dept AND e.salary = m.max_salary;

为什么这样写：子查询 m 只知道每个部门的最高工资是多少，不知道是谁拿到这个工资。回到 emp 表匹配 dept 和 salary，才能拿到员工编号和姓名。

扣分点：只写 `SELECT dept, MAX(salary), name FROM emp GROUP BY dept` 在严格 SQL 中不成立，因为 name 既不是分组属性也不是聚合结果；`ORDER BY salary DESC LIMIT 1` 只会给全公司最高，不是每个部门最高。

题 2｜关系代数：选修了全部课程的学生

Student(SID, Name)，Course(CID, CourseName)，Take(SID, CID)。用关系代数写出“选修了全部课程的学生姓名”。

展开答案与解析
题眼“全部课程”是关系代数除法的强信号。先找出满足“SID 对所有 CID 都出现”的学生，再拿姓名。
AllTakeSID = Π_SID,CID(Take) ÷ Π_CID(Course)
Result = Π_Name(Student ⋈ AllTakeSID)

为什么不是直接连接：普通连接只能表示“选了某些课”，不能表达“对每一门课都选了”。除法正是用来表达“所有/全部/每一个”。

迁移：“参加全部剧目”“购买所有商品”“去过所有城市”在关系代数里都是类似结构：先拿二元关系，再除以全集条件。

题 3｜事务：先画冲突图，再谈串行顺序

调度：r1(B), w1(B), r2(B), w2(B), r3(A), w3(A), r2(A), w2(A), c2, r1(D), w1(D), c3, c1。判断是否冲突可串行化，并写出等价串行顺序或说明原因。

展开答案与解析

B 上的冲突：T1 的 r/w(B) 在 T2 的 r/w(B) 之前，且至少一个写，所以画 T1 -> T2。

A 上的冲突：T3 的 r/w(A) 在 T2 的 r/w(A) 之前，所以画 T3 -> T2。

D 上：只有 T1 操作 D，不产生事务间冲突边。

结论：优先图有 T1 -> T2、T3 -> T2，没有环，所以冲突可串行化。等价串行顺序可为 T1, T3, T2 或 T3, T1, T2，只要 T2 在 T1/T3 之后即可。

扣分点：不要把 commit 顺序当成串行顺序。冲突可串行化看的是冲突操作先后，不是提交先后。

题 4｜恢复：先分 winner / loser

日志中 T1、T2、T3 都 START；T2 在崩溃前 COMMIT，T3 在崩溃前 COMMIT，T1 在崩溃前 COMMIT。若崩溃点在 T1 COMMIT 之后，恢复时三者分别如何处理？若崩溃点提前到 T1 COMMIT 之前，又如何处理 T1？

展开答案与解析

崩溃在 T1 COMMIT 之后：T1、T2、T3 都是已提交事务，属于 winner。恢复时应保证它们的修改被保留；若页面尚未落盘，需要 Redo 或确认已持久化。

崩溃提前到 T1 COMMIT 之前：T1 是未提交事务，属于 loser。恢复时必须 Undo T1 已经执行过的写操作，用日志旧值撤销。

为什么要这样：未提交事务不能留下半截影响，这是原子性；已提交事务不能因为崩溃丢失，这是持久性。

扣分点：不要看到“写了新值”就保留。是否保留先看 commit 状态。

题 5｜规范化：2023A 候选键与 BCNF 起手

R(A,B,C,D,E)，F={ABC -> D, E -> B, AD -> C}。求候选键，并给出一种 BCNF 分解思路。

展开答案与解析
候选键A、E 没出现在任何依赖右部，不能由其他属性推出，所以候选键必须包含 A、E。

AEC 的闭包：AEC 先有 A,E,C；由 E -> B 得 B；有 A,B,C 后由 ABC -> D 得 D，所以 AEC+ = ABCDE。且 A/E/C 任一去掉都推不全，所以 AEC 是候选键。

AED 的闭包：AED 先有 A,E,D；由 E -> B 得 B；由 AD -> C 得 C，所以 AED+ = ABCDE。且 A/E/D 任一去掉都推不全，所以 AED 是候选键。

BCNF 判断：ABC 不能推出 E，不是超码；E 不能推出 A,C,D，不是超码；AD 不能推出 E,B，不是超码。因此 R 不满足 BCNF。

一种分解：可先按 E -> B 分解为 R1(E,B)、R2(A,C,D,E)；再在 R2 中按 AD -> C 分解为 R21(A,D,C)、R22(A,D,E)。得到 {EB, ADC, ADE}。分解顺序不唯一，关键是每一步说明由哪条违反 BCNF 的依赖触发。

高分提醒：题目若继续问保持依赖/无损连接，不能只写“分解完了”。保持依赖要看原 F 是否能由分解后的依赖推出；无损连接要看每步公共属性能否推出某一边。

题 6｜ER：诊所系统先抽实体，再画联系

诊所系统：多名患者和医生；患者看医生形成门诊记录；每次门诊有时间和诊断；门诊会开检查单和/或药品单；检查和药品有名称、价格等信息。请写出 ER 图起手和关系模式转换思路。

展开答案与解析

实体：Patient(pid, name, age)，Doctor(name 或 did, age, dept, title, position)，Visit(visit_id, time, diagnosis)，Exam(exam_name, price, description)，Drug(drug_name, price)。

联系：患者与医生通过 Visit 发生一次门诊。Visit 有自己的编号、时间、诊断，所以更像联系实体/弱化后的业务实体，而不是一条简单连线。

检查/药品：一次 Visit 可以对应多项检查和多种药品，检查/药品也可被多次门诊使用，因此通常用 VisitExam(visit_id, exam_name, exam_time) 和 VisitDrug(visit_id, drug_name) 这种关系表表达 M:N。

高分提醒：ER 题允许说明假设。比如医生是否用姓名作标识，如果现实中姓名可能重名，你可以注明“按题意以姓名为标识，或引入 doctor_id 更稳”。不确定处写清假设比乱画更容易得分。
