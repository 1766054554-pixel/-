1. 先把老题压成五个盒子

软工综合题通常给你一段业务材料。你不能一看到材料就画图，要先分盒子。下面这些盒子来自往年题，不是我凭空编的。

实验上机安排来自 2020：类图、用例图、两张顺序图。题眼是教师、学生、管理员、实验项目、机房、安排、新闻。
选课管理来自 12 级：用例图、课程注册事件流、用户名等价类。题眼是角色功能和基本/异常事件流。
库存管理来自 14 级：用例图/类图、状态图/顺序图、白盒覆盖。题眼是部门职责、商品流通、库存上下限。
在线审稿来自 2016：作者、审稿人、主席、投稿、分配审稿、关闭审稿。题眼是参与者和用例识别。
预订系统来自 2016：游客、售票处、预订、支付、取消、退款、天气自动取消。题眼是类名填空、状态/策略、支付方式。
测试设计来自 12/14/参考答案：等价类、边界值、白盒分支/判定覆盖、测试用例表。
学习顺序：先做本页 18 题客观分流，确认概念边界；再做 5 类材料变式，练“题干 -> 图类型 -> 元素清单”；最后回 软工综合交互卷 做整套。

3. 5 类材料变式：同一段材料问法不同，图也不同

每个材料都按“题眼、该画什么、必备元素、扣分点”训练。你先写自己的骨架，再展开答案。能写出骨架就已经不是裸奔了。

材料 1｜实验上机安排系统：2020 老题换成正式综合题

教师提交课程实验项目、安排实验机房、查询安排结果并打印报表；学生查询实验项目和课程实验安排；管理员管理教师、学生、课程实验和新闻公告；所有用户可修改密码、查看公告，未登录用户登录后成为对应用户。

展开参考骨架
用例图参与者：教师、学生、管理员、未登录用户/用户。用例：登录、修改密码、查看公告、提交实验项目、安排机房、查询安排、打印报表、管理基本信息、管理新闻。参与者在系统边界外，用例用动宾短语。
类图领域类：用户/教师/学生/管理员、课程、实验项目、实验机房、实验安排、新闻公告、报表。关系要标多重性：课程包含实验项目，实验安排关联项目和机房。
顺序图学生查询安排：学生 -> 查询界面 -> 查询服务 -> 实验安排/数据库 -> 返回结果。管理员管理新闻：管理员 -> 新闻管理界面 -> 新闻服务 -> 新闻公告/数据库 -> 返回成功。
扣分点把“课程/机房/新闻”画成参与者；类图不写多重性；顺序图只画“学生 -> 系统”一条线，没有界面/服务/数据对象。

材料 2｜选课管理系统：用例图 + 事件流 + 等价类

学生查询课程、课程注册、查询成绩；教务管理员维护教师/学生/课程信息，开启/关闭课程注册；教师查询课程和选课学生，登记成绩；所有用户登录后使用系统。课程注册有时间窗口、最多 3 门、课程人数 5-30、可试听和改选。

展开参考骨架
用例图学生、教务管理员、教师是核心参与者。登录可作为公共前置用例；课程注册、查询课程、查询成绩、维护信息、开启/关闭注册、登记成绩分别归到对应角色。
事件流基本流：学生登录 -> 进入课程注册 -> 系统显示课程目录 -> 学生选择课程 -> 系统校验人数/上限/时间 -> 保存结果 -> 提示成功。
异常流注册未开启、超过 3 门、课程已满、低于 5 人课程取消、注册关闭后不能改选。异常流要写“系统如何响应”，不是只列异常名。
扣分点事件流写成图名堆砌；异常只写“出错”；登录前置条件和课程注册业务规则漏掉。

材料 3｜库存管理系统：多图分流

计划部制定库存计划和上下限；采购部根据库存制定采购计划并采购；销售部制定出货单；库存部根据采购单入库、根据出货单出库，并记录商品流通。库存低于下限时自动通知计划部。管理员负责授权和注册。

展开参考骨架
用例图参与者：计划部人员、采购部人员、销售部人员、库存部人员、管理员。用例：制定库存计划、制定采购计划、采购、制定出货单、安排入库、安排出库、记录流通、通知采购、授权注册。
类图类：库存计划、商品、采购计划、采购单、出货单、入库记录、出库记录、商品流通记录、部门/工作人员。关系：计划包含商品上下限，入/出库记录改变商品数量。
状态图选一个对象，如“商品库存项”或“采购单”。商品库存项可有正常、低于下限、待采购、已补货等状态。状态图不是把所有部门流程塞进去。
顺序图库存低于下限自动通知：库存监控 -> 库存项 -> 通知服务 -> 计划部/采购计划服务。按时间展示消息。

材料 4｜在线会议审稿：参与者/用例识别

用户注册为作者或审稿人；作者提交稿件、浏览审阅结果；审稿人设置兴趣领域、审阅稿件、列出录用/拒绝稿件；主席是特殊审稿人，可浏览稿件、分配审稿、关闭审稿过程，关闭必须包括列出录用/拒绝稿件。

展开参考骨架
参与者作者、审稿人、会议主席。主席是特殊审稿人，可画成泛化关系。普通“用户”也可作为抽象参与者承接注册/登录。
用例注册、登录、提交稿件、浏览审阅结果、设置兴趣、审阅稿件、列出录用/拒绝稿件、分配稿件、关闭审稿过程。
include关闭审稿过程必须包括列出录用/拒绝稿件，偏 include。提交稿件前的标题、摘要、主题、文件位置校验属于提交稿件内部流程，不一定拆成外部参与者用例。
扣分点把“稿件”“主题类型”“数据库”当参与者；主席和审稿人关系不清；把必须发生和可选扩展混在一起。

材料 5｜预订系统：类图/状态图/策略识别

游客通过售票处预订小木屋或营地，登记信息、计算租赁费用、付款；信用卡支付有 3% 折扣；取消预订按时间收赔偿金；恶劣天气自动取消并全额退款；售票处工作人员查询预订和使用情况。

展开参考骨架
类图Customer、TicketingOfficer、Reservation、ReservationItem、Bungalow、CampSite、Payment、CashPayment、CreditCardPayment、Discount、Rate、NationalPark。支付方式可用继承/策略表达。
状态图选 Reservation：申请中、已批准/已预订、已支付、已取消、已退款、已入住/已结束。取消事件根据时间触发不同赔偿规则。
顺序图游客提出申请 -> 售票员录入 -> 预订服务查询资源 -> 费用服务计算 -> 支付服务付款 -> 创建 Reservation -> 返回确认。
扣分点把支付折扣写成普通备注不建模；取消规则没有状态/条件；类图只列类名不写关系。

4. 测试设计模板：等价类和白盒覆盖最容易拿分

等价类三步 拆输入条件 -> 列有效/无效等价类 -> 用最少测试覆盖每类。比如用户名：长度 8-12、字母数字混合、不能纯字母/纯数字、不能含非法字符。
白盒覆盖三步 画/读控制流 -> 找判断条件 -> 设计输入让每个分支或条件取到要求值。分支覆盖看真假分支，条件覆盖看每个小条件。
故障、错误、失效 故障是代码或设计里的缺陷；错误是缺陷被触发后的内部异常状态；失效是外部用户看到的“不符合需求”。简答题写成“内部原因 -> 运行状态 -> 外部表现”最稳。
题型标准答案结构高频扣分点
等价类输入条件、有效类、无效类、编号、测试输入、预期结果、覆盖编号。只列有效类；无效类重复；测试用例没有预期结果。
边界值围绕最小值、最大值、刚小于、刚大于设计测试。只测中间值，不测边界。
判定/分支覆盖让每个判断的真分支和假分支至少执行一次。把条件覆盖和分支覆盖混用，没说明覆盖了哪条分支。
条件覆盖让每个判断中的每个基本条件都取到真和假。只覆盖了总判断结果，没有覆盖每个小条件。

5. 过关闸门：什么时候去做整套软工卷

客观题闸门18 题至少对 14 题，且能解释错题为什么错。
图题闸门看到材料能先说出：用例图、类图、顺序图、状态图分别回答什么问题。
测试题闸门能独立写一张等价类表，能解释分支覆盖和条件覆盖区别。
下一步通过后做 软工官方五类题型综合交互卷，再做 CodeArts 检查清单。
加油一下：软工其实很适合临时抱佛脚，因为它不是靠玄学灵感，而是靠“题干信号 -> 图类型 -> 元素清单 -> 检查项”。你现在把这条链练熟，考场会明显稳很多。
