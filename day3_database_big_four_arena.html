0. 使用规则：先写流程，不先看答案

第一步写题型：范式、事务、恢复、ER。
第二步写模板第一句：闭包、优先图、Undo/Redo 表、实体联系。
第三步再展开计算或画图，最后看解析补错因。
过关标准：每关至少一题能不看答案写出题眼和第一步。数据库大题后半拿分靠步骤，第一步错了后面再努力也会散。

1. 范式关：候选码是整题的钥匙

FD-1 候选码与 BCNF 判断

题干：关系 R(A,B,C,D,E)，函数依赖 F={ A → B, B → C, CD → E, E → A }。判断 AE 是否为候选码，并判断依赖 A → B 是否违反 BCNF。

展开标准答案与解析

AE 不是候选码。从 AE 出发，E → A 已有 A；A → B 得 B；B → C 得 C。此时 AE+ = {A,B,C,E}，没有 D，不能推出全部属性，所以 AE 不是超码，更不是候选码。

A → B 违反 BCNF。判断 BCNF 看每条非平凡依赖 X → Y 的左边 X 是否为超码。A+ = {A,B,C}，不能推出 D/E，所以 A 不是超码。因此 A → B 违反 BCNF。

常见错因：看到 E → A、A → B、B → C 就以为 AE 很强，但 D 无法推出，缺 D 就不能成为码。范式题一定要写闭包，不要凭感觉。

变式提醒：若题干问 CD 是否为候选码，CD+ 可推出 E，再由 E 推 A，再由 A 推 B，再由 B 推 C，最后有 A/B/C/D/E；还要检查 C 或 D 是否能单独推出全部。

FD-2 BCNF 分解第一刀

题干：R(A,B,C,D)，F={ A → B, B → C }。已知 A+={A,B,C}，A 不是超码。若按 A → B 做 BCNF 分解，第一刀怎么拆？

展开标准答案与解析

第一刀：R1(A,B)，R2(A,C,D)。

模板：对违反 BCNF 的依赖 X → Y，常用分解为 R1 = X ∪ Y，R2 = R - (Y - X)。这里 X=A，Y=B，所以 R1={A,B}，R2={A,C,D}。

为什么 R2 要保留 A：A 是连接两个分解关系的共同属性。若把 A 也拿掉，就无法用 A 把 R1 和 R2 无损连接回来。

下一步：继续检查 R1 和 R2 是否满足 BCNF。不要拆完第一刀就停止，除非题目只问第一步。

2. 事务关：冲突优先图先于结论

TX-1 冲突可串行化

调度：r1(A), w2(A), r2(B), w1(B), w3(A), r3(B)。判断是否冲突可串行化。

展开标准答案与解析

冲突边：A 上 r1(A) 在 w2(A) 前，得 T1 → T2；r1(A) 在 w3(A) 前，得 T1 → T3；w2(A) 在 w3(A) 前，得 T2 → T3。B 上 r2(B) 在 w1(B) 前，得 T2 → T1；w1(B) 在 r3(B) 前，得 T1 → T3。

结论：图中同时有 T1 → T2 和 T2 → T1，形成环，所以该调度不是冲突可串行化。

常见错因：只看同一数据项但忘了“至少一个写”；或把同一事务内部操作也画边。冲突边只在不同事务之间画。

TX-2 两段锁 2PL

事务 T1：lock-X(A), read(A), write(A), unlock(A), lock-S(B), read(B), unlock(B), commit。T1 是否满足 2PL？

展开标准答案与解析

不满足 2PL。两段锁要求一个事务在释放任何锁之后，不能再申请新的锁。T1 在 unlock(A) 之后又执行 lock-S(B)，这违反了 2PL。

第一步模板：不要逐条背锁类型，先找第一次 unlock。第一次 unlock 之前是增长阶段，之后是收缩阶段；收缩阶段不能再 lock。

变式提醒：如果问严格 2PL，还要看写锁是否保持到 commit/abort，目的是避免别人读到未提交数据。

3. 恢复关：先分事务状态，再谈 Undo/Redo

REC-1 日志恢复分流

日志：<T1 start>, <T1, A, 10, 20>, <T2 start>, <T2, B, 30, 40>, <T1 commit>, <T3 start>, <T3, C, 50, 60>, crash。问恢复时哪些事务 Redo，哪些 Undo？

展开标准答案与解析

已提交：T1 已 commit，所以需要 Redo 或确认其结果持久化。

未提交：T2 没有 commit，T3 没有 commit，所以需要 Undo。

恢复动作：Redo T1，把 A 恢复到新值 20；Undo T2，把 B 从 40 撤回旧值 30；Undo T3，把 C 从 60 撤回旧值 50。

常见错因：看到日志值就急着算变量，忘了事务状态。恢复题第一步永远是按 commit 分组。

REC-2 Undo/Redo 的人话解释

题干：简答：为什么系统崩溃后，已经提交的事务要 Redo，未提交的事务要 Undo？

展开标准答案与解析

标准答法：已经提交的事务代表数据库已经向用户承诺其结果有效，为满足持久性，系统崩溃恢复时要确保这些修改最终出现在数据库中，因此需要 Redo。未提交事务在崩溃前尚未完成，不能让半截修改留在数据库中，为满足原子性，需要 Undo 撤销其已经写入的影响。

关键词：提交事务、持久性、未提交事务、原子性、半截修改。

变式提醒：如果题目出现检查点，还要从检查点位置缩小需要扫描和恢复的范围，但“提交 Redo、未提交 Undo”的原则不变。

4. ER 关：业务题先圈实体、属性、联系、基数

ER-1 门诊预约系统抽取

题干：医院有医生、患者和科室。一个科室有多名医生；患者可以预约医生的门诊号源；一次预约记录包含预约时间、状态和挂号费用；医生可以开具多张处方，处方包含多个药品及用量。

展开标准答案与解析

实体：Doctor、Patient、Department、Appointment、Prescription、Drug。

主要属性：Appointment 有 appointmentTime、status、fee；Prescription 有 prescriptionId、createdAt；Drug 有 drugName、specification；Doctor/Patient/Department 各有编号和姓名/名称。

联系与基数：Department 1:N Doctor；Patient 1:N Appointment；Doctor 1:N Appointment；Doctor 1:N Prescription；Patient 1:N Prescription；Prescription 与 Drug 是 M:N，联系属性可包含 dosage、frequency。

转换提示：M:N 的 Prescription-Drug 通常转成中间关系 PrescriptionDrug(prescriptionId, drugId, dosage, frequency)。Appointment 作为实体比单纯联系更稳，因为它有时间、状态、费用等自己的属性。

常见错因：把“预约时间、状态、费用”画成实体；漏掉 M:N 的处方-药品中间表；只画医生和患者，漏科室。

5. 回炉模板：数据库大题后半错题怎么记

范式错题 写：候选码算错在哪一步；哪个依赖左边不是超码；分解第一刀如何写。
事务错题 写：漏了哪条冲突边；是否误把同一事务画边；优先图是否有环。
恢复错题 写：事务状态分错了吗；Undo/Redo 集合是否反了；旧值新值有没有看反。
ER 错题 写：漏了哪个实体/联系/基数；哪个属性不该独立成实体；M:N 是否转中间表。
