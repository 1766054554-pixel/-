0. 使用路线：软工不是背定义，是带着模板查资料

先学 Day1 软工画图从零负责建立语言：用例图、顺序图、类图、状态图、活动图分别回答什么问题。
再做 Day2 2020 三图手把手负责把同一段题干拆成用例图、类图、顺序图。
本页 Day3 本页负责纸质开卷：题眼写到哪里、CodeArts 里怎么下第一笔、图旁文字怎么写、测试题怎么答。
软工考试策略：画图题先拿结构分，再拿说明分。图不一定要和标准答案一模一样，但必须对象合理、关系清楚、箭头方向对、文字说明能自圆其说。考场上先在草稿区列元素，再去 CodeArts Modeling 画，不要边想边乱拖组件。

0.2 软工建模主线：同一段材料，按小问落到不同产物

软工综合题通常给一段系统材料，然后让你做需求分析、概要设计、详细设计。你不能看到“系统”就默认画类图。真正的第一步是先判断小问所在层次，再选择产物：需求题答“系统该做什么”，概要题答“系统怎么分块”，详细题答“对象和方法怎么协作”。

需求分析看外部视角用户、系统边界、系统操作、业务概念。对应十讲里的系统接口、系统操作、领域概念。
概要设计看系统分块模块、包、组件、接口、架构风格。它不该直接陷入某个方法内部代码。
详细设计看对象协作类的属性方法、消息顺序、状态变化、数据持久化。这里才适合设计类图和实现顺序图。
记住这一句：需求说“系统应该做什么”，概要说“系统分成哪些部分”，详细说“对象和方法怎么协作实现”。

2. 图题分流：先判断“它要你回答什么问题”

高频误画：把用例图画成流程图；把顺序图画成“学生 -> 系统”两条线；把类图画成数据库 ER；把状态图画成活动图；把操作合约写成操作步骤。考试最容易扣分的是图类型错位，所以每次下笔前先问：这道小问到底让我回答什么问题？

3. 题干拆解示范：同一句话怎么变成三种图

软工画图最怕“一看题干就画线”。真正稳的做法是先把自然语言拆成四类词：角色、动作、对象、约束。下面用同一个场景演示：同一句话，问法不同，图就不同。

场景： 学生登录实验上机安排系统后，可以查询某课程的实验项目和实验机房安排。 系统根据课程编号查询课程实验项目，再查询该项目对应的实验机房、时间和地点，最后把安排结果显示给学生。
角色：学生 动作：登录、查询、显示 对象：课程、实验项目、实验安排、机房 顺序：输入课程编号 -> 查项目 -> 查安排 -> 返回结果

第一问：如果画用例图，题干在问什么？

用例图不关心“先查项目还是先查机房”，它关心的是：谁在系统外使用系统，系统提供了哪些功能。

从题干抽取：学生是参与者；查询课程实验项目、查询实验机房安排是用例；登录可能是公共用例。

交卷句：本用例图描述学生在实验上机安排系统中的查询功能边界，学生可查询课程实验项目和机房安排。

第二问：如果画顺序图，题干在问什么？

顺序图关心一次具体场景按时间怎么发生。所以要把“系统”拆成界面、控制对象、数据对象，不能只画学生和系统两根线。

从题干抽取：学生发起查询；查询界面接收课程编号；查询控制器协调；课程项目库和安排库返回数据。

交卷句：本顺序图描述学生查询课程实验安排的一次完整交互，时间顺序自上而下。

第三问：如果画类图，题干在问什么？

类图关心静态结构：系统里有哪些长期存在的对象，它们有什么属性，彼此有什么关系。它不关心一次点击流程。

从题干抽取：学生、课程、实验项目、实验安排、机房是候选类；实验安排连接实验项目和机房。

交卷句：本类图抽取了查询安排相关的核心领域类，ExperimentSchedule 作为安排记录关联实验项目与机房。

第四问：如果题目没说图类型怎么办？

用这四个问题判断：

题干在问谁用系统做什么？用例图
题干在问一次操作按时间怎么走？顺序图
题干在问有哪些对象和关系？类图
题干在问某对象状态怎么变，或业务步骤怎么分支？状态图/活动图
临场口诀：谁用什么功能是用例；谁先给谁发消息是顺序；谁和谁长期有关是类图；东西活到哪一步是状态；人和系统按步骤办事是活动。

4. 五类图模板：先画最低得分骨架，再补细节

图/模板第一笔必备元素文字说明模板
用例图画系统边界框，框外列参与者参与者、用例椭圆、系统边界、include/extend/泛化“本图描述 X 系统的功能边界。参与者包括...，主要用例包括...”
顺序图列生命线：参与者、边界对象、控制对象、实体/数据对象生命线、激活条、同步消息、返回消息可选、时间自上而下“本图描述 X 场景的一次交互流程，从用户发起请求到系统返回结果。”
类图先写候选类名，不急着补属性方法类名、必要属性、必要方法、关联、多重性、继承/聚合/组合“本图抽取了系统的核心领域类，类之间关系和多重性如下...”
状态图选一个有生命周期的对象初态、状态、事件/条件、转移、终态“本图描述 X 对象从创建到结束的状态变化。”
活动图列业务步骤，必要时分泳道开始、动作、判断、分支、合并、结束、泳道“本图描述 X 业务流程，泳道区分用户与系统职责。”
操作合约写操作名和前置条件操作、交叉引用、前置条件、后置条件后置条件按“创建对象、属性改变、形成/消除关联”三类写。

include / extend / 泛化：不要只记英文

include 必做公共步骤。比如“提交订单”必须“验证登录”，提交订单 include 验证登录。箭头指向被包含用例。
extend 可选扩展行为。比如“查询安排”之后可以“打印报表”，打印报表 extend 查询安排。箭头指向被扩展的基础用例。
泛化 “是一种”。学生/教师/管理员都是用户，箭头指向更一般的父类或父参与者。
箭头方向一句话：include 箭头指向“必做的公共步骤”；extend 箭头指向“被扩展的基础用例”；泛化箭头指向“更一般的父类/父参与者”。考试不怕你忘英文，怕你把方向画反。

关系人话例子扣分点

5. 测试题模板：软工不只画图，测试也容易拿分

往年资料里等价类、边界值、白盒覆盖反复出现。测试题和画图题一样，也要先分流。老师给的样例里还出现了“故障、错误、失效”的简答题，这类题要背得准，但更要讲清关系。

故障、错误、失效：简答题先写传播链

故障 Fault 藏在软件制品里的缺陷，可以在需求、设计、代码、测试用例中出现。它是“病根”，但不一定每次运行都会暴露。
错误 Error 程序运行时产生了不正确的内部状态，例如变量值错、分支走错、数据结构被破坏。错误是故障被触发后的内部表现。
失效 Failure 系统对外表现不符合需求，例如页面返回错误结果、服务崩溃、超时。失效是用户或外部环境能观察到的问题。
简答第一句话：故障 -> 错误 -> 失效，是从制品缺陷到运行内部状态，再到外部可观察失败的传播链。后面再补一句：一个故障可能导致多个错误，一个错误可能导致失效，也可能被掩盖而不表现为失效。
题眼方法答题模板
输入范围、合法/非法、用户名长度等价类划分列输入条件 -> 有效等价类 -> 无效等价类 -> 编号 -> 测试用例覆盖编号。
最大/最小/刚好超过/刚好不足边界值分析取边界点和边界两侧，如 8-12 位取 7、8、9、11、12、13。
每条语句至少一次语句覆盖设计用例让所有语句执行。
每个判定真假都走到判定/分支覆盖每个 if/while 的真分支、假分支至少一次。
每个条件取真/假条件覆盖对复合条件中的每个小条件分别覆盖 T/F。
所有条件组合条件组合覆盖复合条件的所有 T/F 组合至少出现一次。
所有可能执行路径路径覆盖覆盖控制流图中每条可行路径。
等价类题的写法：不要只写“有效/无效”。必须把输入条件拆成小条件，例如“用户名 8-12 位”和“由字母数字组成”是两个条件；有效类必须完全满足，无效类通常每次违反一个条件，方便定位错误。

6. 场景题卡：先写作答骨架，再看参考答案

题卡 0：综合题三层分流

题目：材料说“学院需要开发实验预约平台，学生预约机房，教师发布实验项目，管理员审核资源安排”。如果小问分别是“做需求分析与建模”“做系统概要设计”“做详细设计”，你各自应该优先输出什么？

查看参考答案

需求分析与建模：优先做用例图、系统顺序图、系统操作合约、概念类图。核心是学生、教师、管理员能做什么，系统操作前后业务对象怎么变化。

系统概要设计：优先做包图、组件图或架构分层说明。可以拆成用户权限、实验项目、预约管理、资源审核、通知、数据持久化等组件，说明职责和接口。

详细设计：优先做设计类图、实现顺序图、状态图、活动图或数据模型。核心是某个用例如何由界面、控制类、实体类协作完成，类有什么属性方法，对象状态如何变化。

迁移以后看到综合题，不要先问“画什么漂亮”，先问“这道小问站在需求、概要还是详细设计层”。层次判断对了，图就不会跑偏。

题卡 1：图类型分流

题目：题干说“学生提交预约后，系统检查时间冲突，若无冲突则保存预约并返回成功提示；若冲突则提示重新选择时间”。如果题目没有指定图类型，你会优先考虑哪两类图？为什么？

查看参考答案

优先考虑顺序图或活动图。若题目关注对象间按时间传递消息，画顺序图：学生、预约界面、预约控制、机房安排/数据库等生命线；若题目关注业务流程和分支，画活动图：提交预约 -> 检查冲突 -> 分支 -> 保存或提示重选。

题眼“提交后系统检查、若...则...”同时有时间顺序和判断分支。

题卡 2：用例图第一笔

题目：实验上机安排系统中，教师可以提交课程实验项目、安排实验机房、查询安排结果、打印报表。学生可以查询实验项目和安排。管理员管理教师、学生、课程实验、新闻公告。请写出用例图起手清单。

查看参考答案

参与者：教师、学生、管理员、未登录用户或普通用户。系统边界：实验上机安排系统。教师用例：提交课程实验项目、安排实验机房、查询安排结果、打印安排报表、修改密码、查看公告。学生用例：查询课程实验项目、查询课程实验安排、修改密码、查看公告。管理员用例：管理教师、管理学生、管理课程实验、管理新闻公告、查询机房时间安排、修改密码、查看公告。未登录用户：登录。

扣分点不要把“课程实验项目、机房、新闻公告”画成参与者，它们是系统内对象或被管理资源。

题卡 3：顺序图生命线

题目：画“学生查询课程实验安排”的顺序图，至少应该列哪些生命线？每条线大概负责什么？

查看参考答案

可列：学生，查询界面，实验安排控制器/服务，课程实验安排数据或数据库。流程：学生发起查询 -> 界面提交查询条件 -> 控制器校验并请求安排数据 -> 数据对象返回安排列表 -> 控制器整理结果 -> 界面显示给学生。

关键顺序图不是“学生 -> 系统”两条线糊过去。至少要体现边界对象、控制对象、实体/数据对象。

题卡 4：类图候选类与关系

题目：从实验上机安排系统中抽类：教师、学生、管理员、课程实验项目、实验机房、实验安排、新闻公告、用户。请写出可能的继承和关联。

查看参考答案

可设 User 为父类，Teacher、Student、Admin 继承 User。Teacher 提交多个 ExperimentProject；ExperimentSchedule 关联 ExperimentProject 和 LabRoom，表示某项目安排在哪个机房、什么时间；Admin 管理 User、ExperimentProject、NewsAnnouncement；Student 查询 ExperimentProject 和 ExperimentSchedule。多重性示例：一个教师可提交多个项目，一个项目通常属于一个教师；一个机房可对应多个安排，一个安排对应一个机房。

提醒类图可以有合理差异，但必须解释关系和多重性。题目明确要求“必要属性和方法、类间关系和数量关系”。

题卡 5：操作合约后置条件

题目：为 arrangeLab(projectId, labId, time) 写系统操作合约后置条件。题意：系统为某课程实验项目安排实验机房和时间。

查看参考答案

后置条件可写：创建了一个新的 ExperimentSchedule 对象；该对象的 projectId、labId、time、status 等属性被设置；该安排对象与对应 ExperimentProject 形成关联；该安排对象与对应 LabRoom 形成关联；若系统需要结果通知，可生成或更新安排结果记录。

模板操作合约后置条件不是“先点击按钮再保存”。它写的是操作完成后系统对象世界发生了什么变化。

题卡 6：include / extend 判断

题目：“用户提交安排前必须登录；查询安排后可以选择打印报表。”分别用 include 还是 extend？箭头指向哪里？

查看参考答案

提交安排 include 登录，因为登录是提交安排前必须执行的公共步骤，箭头指向“登录”。打印报表 extend 查询安排，因为打印是查询后的可选扩展行为，箭头从“打印报表”指向“查询安排”。

方向include 指向被包含的公共用例；extend 指向被扩展的基础用例。

题卡 7：等价类测试表

题目：用户名要求 8-12 位，且只能由英文字母和数字组成。请列有效/无效等价类和至少 4 个测试用例。

查看参考答案

有效等价类：长度 8-12 位；由英文字母和数字组成。无效等价类：长度小于 8；长度大于 12；只含非法字符或包含非法字符；空输入。测试用例示例：Abc12345 有效；Abc1234 无效，长度不足；Abc1234567890 无效，长度超出；Abc123$% 无效，含非法字符。

写法考试最好写成表：输入条件、有效类、编号、无效类、编号、测试用例、预期结果、覆盖编号。

题卡 8：白盒覆盖辨析

题目：判定覆盖、条件覆盖、条件组合覆盖有什么区别？如果一个 if 条件是 A && B，三者分别关注什么？

查看参考答案

判定覆盖关注整个判定表达式的结果真假都至少出现一次。条件覆盖关注每个基本条件 A、B 都至少取到真和假。条件组合覆盖要求 A、B 的所有真假组合都至少出现一次，如 TT、TF、FT、FF。

坑条件覆盖不一定满足判定覆盖，判定覆盖也不一定满足条件覆盖；条件组合覆盖最强但用例数更多。
