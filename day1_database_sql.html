Day 1 数据库 SQL + 关系代数 从零深学讲义

今天先让你第一次真正看懂数据库查询题在说什么。SQL 是写给数据库执行的语言，关系代数是写给阅卷老师看的查询逻辑骨架。这里不假设你知道表、字段、主键、外键、SELECT、WHERE、GROUP BY，也不假设你认识 Π、σ、⋈ 这些符号。所有词都从人话开始。

0. 今天的边界：不求多，只求真的懂

你现在是零基础，所以今天不追求“把数据库查询全部学完”。今天只让你拿到五个最基本的能力：

看到一张表，知道表名、列名、行、主键、外键分别是什么。
知道 SQL 是一种“向数据库提问”的语言，不是神秘咒语。
知道 SELECT、FROM、WHERE、GROUP BY、HAVING 每个词为什么存在。
能把一道中文查询题先翻译成人话步骤，再写很小的 SQL 骨架。
看到关系代数符号，能认出 σ 筛行、Π 选列、⋈ 连接、− 表示差集、÷ 处理“所有/全部”。

如果这四件事不稳，后面连接、嵌套、范式、事务都会像空中楼阁。所以我把 Day2 入口先撤了，先打 Day1 地基。

1. 数据库到底是什么：先别想代码

先想一个最朴素的场景：学校有很多学生，每个学生有学号、姓名、院系、年龄。如果这些信息随便写在纸上，查起来会很痛苦。数据库做的第一件事，就是把信息整理成有规则的表。

SnoSnameSdeptSage
95001李勇CS20
95002刘晨IS19
95003王敏MA18
表像一张规整的名单。上面这张表可以叫 Student，表示学生集合。
列/字段/属性Sno、Sname、Sdept、Sage 都是列，表示每个学生有哪些信息。
行/记录/元组一行就是一个具体学生，比如 95001 这一整行。
值某一行某一列里的具体内容，比如刘晨的 Sdept 是 IS。

你看到课本里的“关系、元组、属性”，先不要害怕。它们基本就是“表、行、列”的数据库版本说法。

为什么数据库不直接说 Excel？

因为数据库不只是存表，还负责查询、约束、权限、事务、恢复等。Excel 更像文件，数据库是管理数据的一整套系统。

2. 主键：一行数据的身份证

如果班里有两个叫“刘晨”的人，只靠姓名就分不清谁是谁。所以数据库需要一个能唯一识别一行的东西，这就是主键。

主键解决的问题这张表里，我怎样准确找到某一行？
学生表常见主键学号 Sno，因为一个学号只对应一个学生。
员工表常见主键员工编号 num，因为一个编号只对应一个员工。
主键不能含糊姓名通常不适合做主键，因为可能重名。

Student(Sno, Sname, Sdept, Sage) 如果 Sno 是主键，意思是：Sno 能唯一确定某个学生。

第一天你只需要记住：主键是“找一行”的最稳入口。

3. 外键：两张表之间的桥

现实里，学生会选课。学生信息放在 Student 表，课程信息放在 Course 表，选课结果放在 SC 表。为什么要分成多张表？因为学生、课程、选课是不同类型的信息。

SC.SnoSC.CnoGrade
95001192
95001285
95002188

SC.Sno 的值不是随便写的，它应该来自 Student.Sno。也就是说，选课表里的学号要能在学生表里找到对应学生。这个“引用别的表主键”的列，就叫外键。

外键解决的问题两张表怎么联系起来？
SC.Sno连接 SC 表和 Student 表。
SC.Cno连接 SC 表和 Course 表。
JOIN 的来源外键经常提示两张表的连接条件。

Student.Sno = SC.Sno Course.Cno = SC.Cno

第一天你只需要记住：主键识别一行，外键连接两张表。

4. SQL 是什么：向数据库提问

SQL 可以先理解为“向数据库提问的语言”。你问数据库：“请从学生表里找出所有计算机系学生的姓名。”数据库需要知道四件事：

从哪里找数据来自哪张表？这里是 Student。
找哪些行只要计算机系学生，所以要筛 Sdept = 'CS' 的行。
显示哪些列题目要姓名，所以只显示 Sname。
要不要排序/分组这道题暂时不需要。

把这四件事翻译成 SQL，就是：

SELECT Sname FROM Student WHERE Sdept = 'CS';
SELECT Sname最后要显示姓名这一列。
FROM Student从学生表里找。
WHERE Sdept = 'CS'只保留院系是 CS 的学生。
分号表示这条 SQL 结束。考试手写时不是最核心，但写上更规范。
为什么 SQL 写起来 SELECT 在前，但你说先想 FROM？

因为写法是给数据库看的，思考顺序是给你自己看的。你自己做题时，必须先知道数据从哪张表来，才知道能筛什么、能显示什么。

4.5 第一次逐行带读：这三行到底各干什么

如果你第一次看到下面这段 SQL，脑子空白很正常。我们不要把它当代码，先把每一行翻译成人话。

SELECT Sname FROM Student WHERE Sdept = 'CS';

FROM Student先看这一行。它回答“去哪里找数据”。这里的意思是：请打开 Student 这张学生表。

为什么先读 FROM？因为如果你不知道打开哪张表，就不知道 Sname、Sdept 这些列从哪里来。

WHERE Sdept = 'CS'这一行回答“哪些行留下来”。意思是：只保留 Sdept 这一列等于 CS 的学生行。

Sdept 是列名，CS 是具体文本值。文本值要加单引号。

SELECT Sname最后读这一行。它回答“最后显示什么”。意思是：通过前面筛选后，只把 Sname 这一列交出来。

所以 SELECT 不是“先去选”，而是“最后交卷显示哪些列”。

零基础常见卡点

很多人一看到 SELECT 就从输出开始想，结果忘记表和条件。你要强行训练自己：中文题 -> 先问 FROM 哪张表 -> 再问 WHERE 要不要筛行 -> 最后才写 SELECT。

5. WHERE：先学会筛行

WHERE 的人话是：“只要满足条件的那些行”。它发生在表还没有分组之前，所以它面对的是一行一行的原始记录。

比较条件年龄小于 20：Sage < 20。
等于条件院系是 CS：Sdept = 'CS'。
范围条件年龄 20 到 23：Sage BETWEEN 20 AND 23。
模糊匹配姓刘：Sname LIKE '刘%'。

SELECT Sname, Sage FROM Student WHERE Sage < 20;

这条 SQL 的人话是：从 Student 表里，只看年龄小于 20 的行，最后显示姓名和年龄。

字符串为什么有引号？

CS、刘% 这种是文本值，SQL 里通常要用单引号包起来。数字 20、23 不需要引号。

5.5 条件不是玄学：先问“这一行能不能判断”

你判断一个条件该不该放 WHERE，最简单的办法是：只看表里的一行，能不能立刻判断真假？

Sage < 20看单个学生这一行就知道年龄是否小于 20，所以是 WHERE。
Sdept = 'CS'看单个学生这一行就知道院系是不是 CS，所以是 WHERE。
Sname LIKE '刘%'看单个学生这一行就知道姓名是否姓刘，所以是 WHERE。
COUNT(*) > 2看单独一行不知道“人数”，必须先把很多行合成一组，所以不是 WHERE。

这是 WHERE/HAVING 的地基。不要先背定义，先问“单独一行能判断吗”。

6. SELECT：不是“先想”，而是“最后交卷”

SELECT 决定最终显示哪些列。题目要学号和姓名，你就 SELECT Sno, Sname；题目要所有列，可以 SELECT *。

SELECT Sno, Sname FROM Student;

这条 SQL 的人话是：从 Student 表中显示学号和姓名两列，不筛行。

SELECT 列名显示指定列。
SELECT *显示所有列。考试里可以用，但大题通常按题目要求写具体列更清楚。

7. 聚合：把很多行算成一个数

现在升级一点。题目不再问“哪些学生”，而是问“有多少学生”“平均成绩是多少”“最高工资是多少”。这时候就要用聚合函数。

COUNT数个数，比如学生人数、选课人数。
SUM求总和，比如总工资。
AVG求平均，比如平均成绩。
MAX/MIN求最大/最小，比如最高工资、最低分。

SELECT COUNT(*) FROM Student;

这条 SQL 的人话是：数 Student 表里一共有多少行，也就是多少个学生。

8. GROUP BY：把行分成几堆再统计

如果题目问“每个系有多少学生”，就不能只数全表人数。你要先按院系分堆：CS 一堆，IS 一堆，MA 一堆。然后每堆各自 COUNT。

SELECT Sdept, COUNT(*) FROM Student GROUP BY Sdept;
FROM Student：先拿到学生表。
GROUP BY Sdept：按院系分堆。
COUNT(*)：每一堆分别数有多少行。
SELECT Sdept, COUNT(*)：最后显示院系和人数。

“每个、各个、按……统计”这类词，通常是在提醒你要 GROUP BY。

9. HAVING：筛分组后的结果

WHERE 筛的是单行，HAVING 筛的是分组后的结果。比如“每个系学生人数超过 2 的系”，人数超过 2 这个条件必须等每个系人数算出来以后才知道。

SELECT Sdept, COUNT(*) FROM Student GROUP BY Sdept HAVING COUNT(*) > 2;
WHERE 管一行年龄小于 20、院系是 CS、姓名姓刘。
HAVING 管一组人数超过 2、平均成绩大于 80、最高工资超过 10000。
为什么不能 WHERE COUNT(*) > 2？

因为 WHERE 执行时还没有分组，COUNT(*) 这个“每组人数”还不存在。先 GROUP BY，才有每组的 COUNT。

9.5 第二次逐行带读：分组题怎么从中文走到 SQL

现在看这句中文：统计每个系学生人数超过 2 的系名。我们慢慢拆，不跳步。

先找统计对象“学生人数”说明要数行数，所以需要 COUNT(*)。
再找分组标准“每个系”说明按 Sdept 分堆，所以 GROUP BY Sdept。
再找筛选条件“人数超过 2”不是单行条件，是每组算完人数后筛，所以 HAVING COUNT(*) > 2。
最后找输出题目要系名，所以 SELECT Sdept。
SELECT Sdept FROM Student GROUP BY Sdept HAVING COUNT(*) > 2;

FROM Student先打开学生表。

GROUP BY Sdept把学生按院系分成几堆。

HAVING COUNT(*) > 2每堆数人数，只保留人数超过 2 的堆。

SELECT Sdept最后显示留下来的系名。

这就是你以后做分组题的固定路线：统计什么 -> 按什么分组 -> 分组后筛什么 -> 输出什么。

10. 今日唯一真题风格预告：回表，不要求你现在完全掌握

下面这个题来自 2023A 风格。它比前面难很多，所以今天只理解“为什么不能一步完成”，不要求你熟练写。

题干：查找各个部门薪水最高的员工编号、姓名和部门号。

先看一张员工表：

numnamedeptsalary
1张三A8000
2李四A9000
3王五B7000
4赵六B9500

如果先按部门分组，只能稳定得到：

A 部门最高工资 9000 B 部门最高工资 9500

但题目要员工编号和姓名。最高工资数值本身不是员工姓名，所以要回到原来的 emp 表，找“部门相同，并且工资等于该部门最高工资”的那一行。这一步叫回表。

先求每个部门的最高工资。
把结果当成一张临时小表。
再和 emp 原表连接。
匹配条件：部门相同，工资等于最高工资。
今天要不要背这段完整 SQL？

不急。今天只要你能解释“为什么要回表”。完整 SQL 后面再练。零基础阶段先把原因讲清楚，别先背长句子。

10.5 中文题干翻译词典：考试不会直接喊 SQL 关键词

考试题通常不会说“请你使用 GROUP BY 和 HAVING”。它会用中文绕一层。你要练的是把中文信号翻译成 SQL 部件。

“查询/列出/找出……”通常是在问最终输出什么，对应 SELECT 后面的列。
“从学生表/员工表……”通常是在提示数据来源，对应 FROM。
“年龄小于/姓名为/院系是……”单独看一行就能判断，对应 WHERE。
“每个/各个/按……统计”说明要分堆，对应 GROUP BY。
“人数/总数/平均/最高/最低”说明要把多行算成一个数，对应 COUNT/SUM/AVG/MAX/MIN。
“人数超过/平均大于/最高低于”如果是分组算完后的条件，对应 HAVING。
“同时显示另一张表的信息”常常要 JOIN，因为信息分散在多张表。
“对应的员工姓名/完整信息”如果前面只算出了最大值，常常要回表。

你可以把它当成考场翻译器。先翻译题干，再写 SQL，别反过来硬凑语法。

10.6 零基础常见误区：错在这里很正常，但要尽早修

误区 1：SELECT 写在前，所以先想 SELECT做题时先想 FROM。没有数据来源，就没有后面的筛选和输出。
误区 2：看到条件就全放 WHERE先问“单独一行能判断吗”。能判断放 WHERE，必须先分组计算才知道就放 HAVING。
误区 3：GROUP BY 后随便 SELECT 普通列分组后，每组里可能有很多行，普通列会变得不确定。
误区 4：背长 SQL，不理解为什么零基础阶段先会解释原因，再背模板。解释不出来，考试稍微变形就会断。

10.7 SQL 题卡：先写骨架，再看逐行解析

这里不是选择题，也不是让你背答案。每题先写三件事：题眼、SQL 骨架、人话解释。写不完整也没关系，重点是从中文走到 SQL。

题 1：单表筛行

已知 Student(Sno, Sname, Sdept, Sage)。查询年龄小于 20 的学生姓名和年龄。

看标准答案和逐行解析

标准答案：

SELECT Sname, Sage FROM Student WHERE Sage < 20;

题眼：“年龄小于 20”看单个学生这一行就能判断，所以放 WHERE。

逐行读：FROM Student 打开学生表；WHERE Sage < 20 留下年龄小于 20 的行；SELECT Sname, Sage 最后显示姓名和年龄。

常见坑：不要写成 SELECT Sname FROM Student WHERE Sage < 20 如果题目还要年龄；输出列要跟题目一致。

题 2：LIKE 模糊匹配

已知 Student(Sno, Sname, Sdept, Sage)。查询所有姓“刘”的学生学号和姓名。

看标准答案和逐行解析

标准答案：

SELECT Sno, Sname FROM Student WHERE Sname LIKE '刘%';

题眼：“姓刘”不是等于某个完整姓名，而是姓名以“刘”开头，所以用 LIKE '刘%'。

% 的意思：百分号表示后面可以接任意字符。'刘%' 可以匹配刘晨、刘伟、刘某某。

常见坑：文本要加单引号；不要写 Sname = '刘%'，等号不会把 % 当通配符。

题 3：分组统计

已知 Student(Sno, Sname, Sdept, Sage)。统计每个系的学生人数，显示系名和人数。

看标准答案和逐行解析

标准答案：

SELECT Sdept, COUNT(*) AS student_count FROM Student GROUP BY Sdept;

题眼：“每个系”提示按 Sdept 分组；“人数”提示 COUNT(*)。

逐行读：FROM Student 打开学生表；GROUP BY Sdept 按系分堆；COUNT(*) 数每堆几行；SELECT 显示系名和人数。

常见坑：如果 SELECT 里显示 Sdept 和 COUNT(*)，通常 Sdept 要出现在 GROUP BY 里。

题 4：HAVING 筛组

已知 Student(Sno, Sname, Sdept, Sage)。查询学生人数超过 2 的系名和人数。

看标准答案和逐行解析

标准答案：

SELECT Sdept, COUNT(*) AS student_count FROM Student GROUP BY Sdept HAVING COUNT(*) > 2;

题眼：“人数超过 2”是分组统计以后才知道的条件，所以用 HAVING，不用 WHERE。

为什么不是 WHERE：WHERE 看单独一行时执行，那时候还没有每个系的人数 COUNT(*)。

考场自查：凡是条件里出现 COUNT/AVG/MAX/MIN 这类聚合结果，大概率要放 HAVING。

题 5：两表连接的第一步

已知 Student(Sno, Sname, Sdept) 和 SC(Sno, Cno, Grade)。查询选修了课程 C1 的学生姓名和成绩。

看标准答案和逐行解析

标准答案：

SELECT Student.Sname, SC.Grade FROM Student, SC WHERE Student.Sno = SC.Sno AND SC.Cno = 'C1';

题眼：姓名在 Student 表，成绩和课程号在 SC 表，所以必须用两张表。两张表通过 Sno 对齐。

逐行读：FROM Student, SC 先拿两张表；WHERE Student.Sno = SC.Sno 把同一个学生的两边信息对上；AND SC.Cno = 'C1' 只保留选 C1 的选课记录；SELECT 输出姓名和成绩。

常见坑：只写 FROM SC 就拿不到 Sname；只写 FROM Student 就拿不到 Grade 和 Cno。信息在哪张表，决定 FROM 要不要加表。

10.8 Day 1 第二课：关系代数符号识字课

你刚刚学的是 SQL。SQL 像一段写给数据库执行的句子；关系代数像一张写给人看的“查询流程图”。考试会让你在这两种语言之间互相翻译，所以这里不能只认识 Π、σ 的名字，必须知道它们长什么样、读什么、在题里放哪一层。

先纠正一个危险误区

关系代数里的投影通常写大写希腊字母 Π，不是普通英文 P；选择通常写小写希腊字母 σ，不是英文 o。你在考卷上看到它们，第一反应要是“哦，Π 选列，σ 筛行”，而不是“这是什么鬼符号”。

符号长相名称人话动作SQL 对应考试写法
σ 选择 selection 筛行：留下满足条件的记录 WHERE σSdept='CS'(Student)
Π 投影 projection 选列：最后只保留题目要求的字段 SELECT ΠSname,Sage(Student)
× 笛卡尔积 product 暴力两两组合两张表的行 FROM R, S 的原始组合 R × S
⋈ 连接 join 按公共属性或条件把表拼起来 JOIN / WHERE 连接条件 Student ⋈ SC
∪ 并 union 合并两个同类型结果 UNION A ∪ B
− 差 difference 在 A 里但不在 B 里 NOT IN / NOT EXISTS 的集合味道 ΠSno(Student) − ΠSno(SC)
÷ 除 division 处理“所有、全部、每一个” 双重 NOT EXISTS / COUNT(DISTINCT) ΠSno,Cno(SC) ÷ ΠCno(Course)
ρ 更名 rename 给关系或属性改名，避免同名冲突 AS / 表别名 ρE(Employee)

Day1 先把前六个练熟：σ、Π、×、⋈、∪、−。看到“所有/全部/每一门”再把 ÷ 叫出来。ρ 是处理自连接或重名属性时的辅助工具，先知道它是“改名”即可。

从 SQL 翻译到关系代数：顺序其实很固定

关系代数表达式通常是从里面往外看。里面先拿表，外面逐层筛行、选列。你可以把它理解成“SQL 逻辑顺序的符号版”。

FROM R, S先拿两张表，对应 R × S 或连接。
WHERE 条件筛掉不符合条件的行，对应 σ条件(...)。
SELECT 列最后只保留输出列，对应 Π列(...)。
没有/所有“没有”常用差集 −；“所有”常用除法 ÷。

2023 风格例子：

SELECT A, B, D FROM R, S WHERE R.C = S.C;

把它翻译成关系代数：

ΠA,B,D( σR.C = S.C( R × S ) )
最里面：R × S，先把 R 和 S 的行暴力组合，这是 FROM R, S 的关系代数底层含义。
中间：σR.C = S.C(...)，只留下 R.C 和 S.C 相等的组合，这就是 WHERE 条件。
最外面：ΠA,B,D(...)，最后只输出 A、B、D 三列，这就是 SELECT。

你也可以写成带条件连接的版本：ΠA,B,D(R ⋈R.C=S.C S)。如果老师给的是笛卡尔积 + 选择的风格，就按前一种写；如果讲义常用连接符，就按后一种也能表达同一逻辑。

真题风格练习：选修 Database 的学生姓名

已知：Student(SID, Name, Age)，Course(CID, CourseName, Credit)，Take(SID, CID)。请用关系代数写出：找选修了课程名为 “Database” 的学生姓名。

看标准答案和逐层解析

第一步：输出列。题目最后要学生姓名，所以最外层一定是 ΠName(...)。这一步对应 SQL 的 SELECT Name。

第二步：筛选条件。课程名为 Database，所以中间需要 σCourseName='Database'(...)。这一步对应 SQL 的 WHERE CourseName = 'Database'。

第三步：需要三张表。Name 在 Student；CourseName 在 Course；学生和课程的选课关系在 Take。所以必须把 Student、Take、Course 连起来。

自然连接简写：

ΠName( σCourseName='Database'( Student ⋈ Take ⋈ Course ) )

更严格的条件连接写法：

ΠName( σCourseName='Database'( (Student ⋈Student.SID=Take.SID Take) ⋈Take.CID=Course.CID Course ) )

考试怎么选写法：如果三张表的同名列 SID、CID 很清楚，写自然连接简写通常够；如果担心同名属性不明确，就写带条件连接，更不容易被扣“连接条件不清”。

关系代数最容易丢分的 6 个坑

坑 1：把 Π 和 σ 写反Π 是选列，σ 是筛行。记忆：Π 像一扇门，最后只让几列出门；σ 像筛子，筛掉不合格的行。
坑 2：最外层忘了投影题目问姓名，你中间连接筛选都对，但最后没写 ΠName，就等于交了一整张大表。
坑 3：跨表不连接Name 在 Student，CourseName 在 Course，中间没有 Take 就搭不上桥。关系代数题也要先问信息在哪几张表。
坑 4：“没有”和“所有”混用没有选课是差集；选修所有课程是除法。一个是否定排除，一个是覆盖全集，题眼完全不同。
坑 5：只背符号，不会读式子看到 ΠName(σCourseName='Database'(...)) 要会读成：最后输出姓名，中间只留 Database 课程相关记录。
坑 6：把 SQL 混进关系代数题目要求关系代数时，不要写 SELECT/FROM/WHERE；要把它们翻译成 Π/×/σ 或连接。

关系代数小测 1：认符号

不看表格，写出下面符号的人话动作：σ、Π、×、⋈、−、÷。

看参考答案

σ 筛行；Π 选列；× 笛卡尔积，暴力组合；⋈ 连接，按共同属性或条件拼表；− 差集，A 有 B 没有；÷ 除法，处理所有/全部/每一个。

关系代数小测 2：从中文到骨架

已知 Student(Sno, Sname)，SC(Sno, Cno)。写出“没有选修任何课程的学生姓名”的关系代数思路。

看参考答案

如果只要学号，最干净是：

ΠSno(Student) − ΠSno(SC)

但题目要姓名，所以可以先求没选课的学号，再和 Student 取姓名：

ΠSname( Student ⋈ (ΠSno(Student) − ΠSno(SC)) )

人话：全体学生里去掉至少在 SC 出现过的学生，剩下就是没有选课的人；最后回 Student 拿姓名。

11. 主观训练：先写人话，再看答案

下面的输入框会自动保存在本机。你现在写得幼稚也没关系，关键是把词变成自己的话。

训练 1

用自己的话解释：表、列、行分别是什么？请用学生表举例。

参考答案

表是一类数据的集合，比如 Student 表保存学生信息；列是每个学生都有哪些字段，比如 Sno、Sname、Sdept；行是一个具体学生的记录，比如 95001、李勇、CS、20 这一整行。

训练 2

主键和外键分别解决什么问题？为什么外键会影响 JOIN？

参考答案

主键解决“一行怎么唯一识别”的问题，比如学号唯一确定一个学生。外键解决“两张表怎么联系”的问题，比如 SC.Sno 引用 Student.Sno。JOIN 要写两张表如何对齐，外键通常就是连接条件来源。

训练 3

题干说“统计每个系学生人数超过 2 的系名”，你会怎么拆？先不用追求 SQL 完美。

参考答案

“每个系”说明按 Sdept 分组；“学生人数”说明要 COUNT(*)；“超过 2”是分组后的条件，所以用 HAVING COUNT(*) > 2；最后输出系名 Sdept。

训练 4

为什么“各部门最高工资员工”不能只 GROUP BY dept 后 SELECT name？

参考答案

GROUP BY dept 后，每个部门会变成一组。MAX(salary) 能得到这个组里的最高工资值，但 name 不是分组字段，也不是聚合结果，数据库无法稳定知道你要哪个员工姓名。要拿员工编号和姓名，必须回到 emp 原表匹配那一行。

训练 5

看到一条 SQL 时，为什么建议你先读 FROM，而不是先读 SELECT？

参考答案

因为 FROM 决定数据来自哪张表。只有先知道打开哪张表，才知道有哪些列可以筛选和输出。SELECT 写在前面只是 SQL 的书写格式；做题理解时应先想 FROM，再想 WHERE/GROUP BY/HAVING，最后想 SELECT 输出什么。

训练 6

用自己的话读出这个关系代数表达式：ΠSname(σSdept='CS'(Student))。它对应哪条简单 SQL？

参考答案

从里面往外读：先打开 Student 表；σSdept='CS' 表示只留下院系为 CS 的学生行；ΠSname 表示最后只输出学生姓名。对应 SQL 是：SELECT Sname FROM Student WHERE Sdept = 'CS';
