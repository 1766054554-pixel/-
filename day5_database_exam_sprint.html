1. 最后 90 分钟：按这个顺序抱

时间做什么目标
0-15 分钟看 SQL 题眼：连接、分组、HAVING、全部/没有、最高/最早、插删改。中文题干能翻译成 SQL 骨架。
15-25 分钟看关系代数：选择 σ、投影 Π、连接 ⋈、差 −、除法 ÷。遇到“全部/所有”能想到除法或双重 NOT EXISTS。
25-45 分钟看范式：闭包、候选键、部分依赖、传递依赖、BCNF 分解。能写候选键和分解第一步。
45-65 分钟看事务：冲突边、优先图、2PL、严格 2PL、可恢复。能从读写序列画边和判断有环。
65-78 分钟看恢复：提交事务保留/重做，未提交事务撤销。Undo/Redo 不乱。
78-90 分钟看 ER：实体、属性、多值属性、弱实体/联系实体、M:N 拆表。能把自然语言落成关系模式。

2. 题眼速查：看到这些词，脑子立刻切模板

SQL：全部/所有/每一个 优先想两种写法：GROUP BY ... HAVING COUNT(DISTINCT ...)=总数，或双重 NOT EXISTS。如果是关系代数，优先想到除法。
SQL：没有/从未/不学 想 NOT EXISTS 或差集。不要用 != 硬写，因为“不等于某一行”不代表“不存在任何一行”。
SQL：至少两门/三门以上 想 GROUP BY 学生 HAVING COUNT(DISTINCT 课程)>=2/3。如果题目要求姓名，最后回学生表。
SQL：最高/最早/最多 想“先聚合得到 max/min/count，再回表取完整信息”。只写 MAX 通常拿不到姓名或完整元组。
范式：主键是 AB，A->C 这是部分依赖信号：非主属性 C 依赖候选键的一部分 A，所以不满足 2NF。
事务：读写交错 同一数据项、不同事务、至少一个写，就画冲突边。优先图有环就不是冲突可串行化。
恢复：START 但无 COMMIT 这是未提交事务，崩溃后要撤销；有 COMMIT 的事务要保证结果保留，必要时重做。
ER：一个客户多个电话 多值属性不要硬塞一个字段，通常单独建电话表：客户电话(客户ID, 电话)。

3. 混合小测：真题风味，不靠猜

题 1：SQL “不学某人学过的课程”

07:00

表：S(S#, SNAME, AGE, SEX)，SC(S#, C#, GRADE)，C(C#, CNAME, TEACHER)。写 SQL：检索 WANG 同学不学的课程号和课程名。

答案与解析 
SELECT c.C#, c.CNAME
FROM C AS c
WHERE NOT EXISTS (
  SELECT *
  FROM S AS s, SC AS sc
  WHERE s.S# = sc.S#
    AND sc.C# = c.C#
    AND s.SNAME = 'WANG'
);
题眼“WANG 不学的课程”要对每一门课程问：是否不存在一条 WANG 选了这门课的记录。
坑不能写 sc.C# != c.C#，因为只要 WANG 学过别的课，就会产生很多“不等于”行，逻辑会错。

题 2：SQL “所有学生都选修的课程”

08:00

同样三张表。写 SQL：检索全部学生都选修的课程号与课程名。

答案与解析 
SELECT c.C#, c.CNAME
FROM C AS c
WHERE NOT EXISTS (
  SELECT *
  FROM S AS s
  WHERE NOT EXISTS (
    SELECT *
    FROM SC AS sc
    WHERE sc.S# = s.S#
      AND sc.C# = c.C#
  )
);
翻译不存在一个学生，使得这个学生没有选这门课。双重否定就是“所有学生都选了”。
替代如果无重修，也可按课程分组，要求选这门课的人数等于学生总数。

题 3：SQL “成绩加分但不能超过 100”

08:00

把物理课成绩都加 20 分，但最终不能超过 100。假设课程表 C 中课程名为 CNAME，选课表 SC 中成绩为 GRADE。

答案与解析 
UPDATE SC
SET GRADE = GRADE + 20
WHERE C# IN (
  SELECT C#
  FROM C
  WHERE CNAME = '物理'
);

UPDATE SC
SET GRADE = 100
WHERE GRADE > 100
  AND C# IN (
    SELECT C#
    FROM C
    WHERE CNAME = '物理'
  );
题眼“但不能超过 100”是边界条件。先加分后截断，是 SQL 专题里常见写法。
坑第二句也要限定物理课，否则可能把别的课原本大于 100 的异常数据也改掉。

题 4：关系代数 “演员参加全部剧目”

07:00

Actor(actor_id, name, year_born)，Play(play_id, title, author, year_written)，Role(actor_id, character_name, play_id)。写关系代数：找参加过所有剧目的演员姓名。

答案与解析 
Π_name (
  Actor ⋈ (
    Π_actor_id, play_id(Role) ÷ Π_play_id(Play)
  )
)
题眼“所有剧目”是除法最典型信号。被除关系必须包含 actor_id 和 play_id，除数只包含 play_id。
步骤先算出满足条件的 actor_id，再和 Actor 连接取 name。不要一开始就投影 name，否则除法需要的 actor_id 会丢。

题 5：外键级联判断

07:00

Take_course(courseID REFERENCES Course(courseID) ON DELETE SET NULL, stuID REFERENCES Student(stuID) ON DELETE CASCADE)。判断：删除 Course 中某门课时，Take_course 中相关 courseID 可能被置 NULL；删除 Take_course 中一行会级联删除 Student 中的学生。

答案与解析

第一句正确。Course 是父表，Take_course.courseID 是引用它的外键。删除 Course 中被引用的课程时，子表 Take_course 的 courseID 按 ON DELETE SET NULL 置空。

第二句错误。ON DELETE CASCADE 的方向是“父表 Student 被删 -> 子表 Take_course 相关行被删”，不是删除子表 Take_course 反过来删父表 Student。

口诀外键动作看父表被删时，子表怎么办。不要把方向想反。

题 6：2NF 部分依赖

08:00

R(A,B,C,D,E)，主键为 (A,B)，函数依赖有 (A,B)->CDE，A->C。判断 R 是否满足 2NF，并给出一个分解方向。

答案与解析

R 不满足 2NF。因为 C 是非主属性，却依赖于候选键 (A,B) 的真子集 A，即存在部分函数依赖 A->C。

分解方向：把部分依赖单独拿出来，可分为 R1(A,C) 和 R2(A,B,D,E)。R1 保存 A->C，R2 保存原主键对其他属性的依赖。

题眼组合主键 + 非主属性依赖主键一部分 = 违反 2NF。
别混3NF 重点是传递依赖；BCNF 重点是每个非平凡 FD 的左边都必须是超键。

题 7：候选键与 BCNF

10:00

R(A,B,C,D,E,F)，函数依赖 F={A->BC, D->AF}。求候选键，并判断是否 BCNF。

答案与解析

候选键：D。因为 D->AF，A->BC，所以 D+ = {D,A,F,B,C}，再加上 D 自己，得到 A,B,C,D,F；注意 E 没有被任何依赖推出，所以所有候选键必须包含 E。于是 DE+ = {D,E,A,F,B,C}，覆盖全体属性，候选键是 DE。

BCNF：不满足。A->BC 中 A 不是超键；D->AF 中 D 也不是超键，因为缺 E。因此违反 BCNF。

分解方向：可先按 A->BC 分解为 R1(A,B,C) 和 R2(A,D,E,F)，再在 R2 中按 D->AF 分解为 R21(D,A,F) 和 R22(D,E)。

关键E 没出现在任何右部，说明 E 不能被推出，候选键必须带 E。这是候选键题的高频隐藏点。

题 8：冲突可串行化

10:00

调度：R1(A), W1(A), R2(A), R2(B), R1(B), W1(B)。判断是否冲突可串行化，并说明理由。

答案与解析

A 上有 W1(A) 在 R2(A) 前，得到边 T1 -> T2。B 上有 R2(B) 在 W1(B) 前，得到边 T2 -> T1。优先图出现环 T1 -> T2 -> T1，因此该调度不是冲突可串行化。

题眼同一数据项、不同事务、至少一个写，才画冲突边。
步骤先按数据项分组看 A、B，不要按整串从左到右乱扫。

题 9：恢复 Undo / Redo

10:00

日志：<START T> <T,A,10,11> <START U> <U,B,20,21> <T,C,30,31> <U,D,40,41> <COMMIT U>，随后崩溃。恢复时 T 和 U 分别怎么处理？

答案与解析

U 有 COMMIT，是已提交事务，恢复后应保证它的修改 B:20->21、D:40->41 生效；如果数据库页还没落盘，就需要 redo。

T 没有 COMMIT，是未提交事务，恢复后不能保留它的修改，应撤销 A:10->11、C:30->31，把 A、C 恢复到旧值 10、30。

核心恢复题先分 winner/loser：有 COMMIT 是 winner，无 COMMIT 是 loser。
别乱不要被 old/new value 绕晕。未提交用 old value 撤销，已提交用 new value 保证结果。

题 10：银行 ER 到关系模式

12:00

银行系统：客户有身份证、姓名、一个或多个电话、职业；账户由账号和所属支行标识，有余额和账户类型；一个账户属于一个或多个客户，一个客户可有多个账户；账户属于且只属于一个支行；员工属于一个支行；员工可作为多个客户的联系人，每个客户至多一个联系人。给出关系模式骨架。

答案与解析 
客户(客户ID, 姓名, 职业, 联系员工ID)
客户电话(客户ID, 电话)
支行(支行代码, 地址, 员工数)
员工(员工ID, 姓名, 工资, 支行代码)
账户(支行代码, 账号, 余额, 账户类型)
拥有账户(客户ID, 支行代码, 账号)
多值属性一个客户多个电话，拆成 客户电话(客户ID, 电话)，不要在客户表里写 电话1/电话2。
M:N客户和账户是多对多，必须建中间表 拥有账户。
联系员工每个客户至多一个联系人，所以可在客户表放 联系员工ID；员工可联系多个客户。
