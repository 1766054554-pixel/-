1. 10 秒分流：题干一出现，你先做什么

题干出现立刻归类第一笔写什么别急着做什么
R(A,B,C,D,E), F={...}, 候选键函数依赖写“先找不在右部的属性，再求闭包”。别凭感觉猜键。
最高范式、3NF、BCNF、分解规范化先列候选键/主属性，再逐条检查 X->Y。别直接背“左边不是键就拆”，3NF 还要看右边是否主属性。

lock/unlock, 2PL锁协议找每个事务第一次 unlock。别只数有没有 lock，要看释放后是否又加锁。

2. 五套得分模板：考场照这个骨架写

模板 A：候选键与闭包

写作顺序
1. 找右部没出现的属性，它们通常必须在候选键中。
2. 从这些属性出发补 1 个或几个属性，求 X+。
3. X+ 包含全部属性则是超码。
4. 删除任一属性都不能推出全部属性，才是候选键。
标准句：因为 A、E 不出现在任何依赖右部，不能由其他属性推出，所以候选键必须包含 A、E。计算 AEC+ / AED+ 均为 ABCDE，且满足最小性，因此候选键为 AEC、AED。

模板 B：3NF / BCNF 判断

3NF
对每条非平凡依赖 X -> A，若 X 是超码，或 A 是主属性，则不违反 3NF。主属性就是出现在某个候选键里的属性。
BCNF
对每条非平凡依赖 X -> Y，X 必须是超码。只要有一条左边不是超码，就不满足 BCNF。
最容易丢分：BCNF 比 3NF 严。3NF 可能因为右边是主属性而放过，BCNF 不放过。

模板 C：BCNF 分解

若 X -> Y 违反 BCNF： R1 = X ∪ Y R2 = R - (Y - X) 然后分别检查 R1、R2，直到每个关系都满足 BCNF。

考场写法要带原因：不是“我把表拆成这样”，而是“因为某条依赖违反 BCNF，所以按这条依赖分解”。

模板 D：冲突可串行化和 2PL

冲突边 不同事务 + 同一数据项 + 至少一个写。谁先出现，就从谁指向后出现的事务。
可串行化 画优先图。无环则冲突可串行化；有环则不可冲突串行化。
2PL 第一次 unlock 后不能再 lock 新数据项。
严格 2PL 写锁通常保持到 commit/abort，可避免级联回滚。

模板 E：恢复与 ER 图

Undo/Redo 未提交事务 Undo，已提交但可能未落盘的事务 Redo 或确认。Undo 用旧值，Redo 用新值。
ER 图 实体是要长期记录的对象；属性是对象信息；联系是对象之间的动作或关系；基数写 1:1、1:N、M:N。

3. 常见错因：你考场最可能掉的分

错法为什么危险修正动作
候选键只写答案，不写闭包老师看不出你会推导，步骤分没了。至少写 2 个闭包：AEC+、AED+。
把“超码”和“候选键”混用超码可多属性冗余，候选键必须最小。先证明能推出全部，再证明去掉任一属性不行。
BCNF 分解没有说明由哪条依赖触发像随便拆表，容易不给过程分。每一步写“因 X->Y 且 X 非超码，故分解”。
冲突图把不同数据项也画边图里会多出假环，结论错误。逐项检查 A、B、C，不同数据项跳过。
2PL 只看锁是否成对释放2PL 的核心不是成对，而是释放后不能再申请。圈第一次 unlock，往右扫有没有 lock。
恢复题先算数值没分 Undo/Redo，后面数值容易全错。先列 Undo 集合和 Redo 集合。
ER 图漏联系属性和 M:N 中间关系数据库设计题常靠这些拉开分差。看到“记录、订单、选课、包含”优先考虑实体化联系。

4. 限时题卡：先写，再展开

点击开始只是给你压力感；真正重要的是先写骨架，别边看答案边觉得“我会”。

题 1：候选键和 BCNF 连环题

07:00

设 R(A,B,C,D,E)，F={AB->C, C->D, D->B, E->A}。求一个候选键，并判断 R 是否满足 BCNF。

评分点和解析
2 分右部出现 C、D、B、A，只有 E 不在右部，所以候选键必须包含 E。
3 分E+：E->A，得到 EA；还不能推出 B/C/D。尝试 EB+：有 E、B、A；AB->C 得 C；C->D 得 D，所以 EB+=ABCDE。EC+：E->A，C->D，D->B，再有 AB->C，也能推出全部。ED+：E->A，D->B，AB->C，也能推出全部。候选键可写 EB、EC、ED。
2 分BCNF 判断要逐条看左部是否超码。AB+ 可推出 C、D，但不能推出 E，所以 AB 不是超码，AB->C 违反 BCNF。C、D、E 的左部也分别检查；只要已有一条违反，就可判定不满足 BCNF。

迁移提醒：这题故意不是 2023A 原题，训练你别背 AEC/AED。第一反应仍然是“找不在右部的属性 + 求闭包”。

题 2：BCNF 分解写过程

08:00

设 R(A,B,C,D)，F={A->B, B->C, C->D}。给出一个 BCNF 分解。

评分点和解析
2 分A+ = ABCD，所以 A 是候选键。B+ = BCD，不含 A；C+ = CD，不含 A/B；因此 B、C 不是超码。
3 分B->C 违反 BCNF，C->D 也违反。也可以先用 A->B 分解，但 A 是原关系的键，A->B 不违反 BCNF，通常不作为必须拆的触发依赖。
3 分稳妥分解路径：先按 C->D 得 R1(C,D)、R2(A,B,C)。R1 中 C 是键，满足 BCNF。R2 中仍有 A->B、B->C；A 是 R2 的键，但 B 不是超码，所以 B->C 违反 BCNF。再按 B->C 分解 R2，得到 R21(B,C)、R22(A,B)。最终可写 {CD, BC, AB}。

高分提醒：BCNF 分解答案不唯一，但每一步必须能解释“当前关系里哪条依赖仍成立且违反 BCNF”。如果不确定投影依赖，考试先选择直观、能保留依赖的链式分解。

题 3：冲突优先图

06:00

调度 S：r1(X), w2(X), r2(Y), w3(Y), w1(Y), r3(X)。写冲突边并判断是否冲突可串行化。

评分点和解析
3 分X 上：r1(X) 与 w2(X) 冲突，T1->T2；w2(X) 与 r3(X) 冲突，T2->T3。
3 分Y 上：r2(Y) 与 w3(Y) 冲突，T2->T3；r2(Y) 与 w1(Y) 冲突，T2->T1；w3(Y) 与 w1(Y) 冲突，T3->T1。
2 分图中有 T1->T2->T1 的环，也有 T1->T2->T3->T1，因此不可冲突串行化。

题 4：2PL 和严格 2PL

05:00

T1: lock-X(A), write(A), lock-S(B), read(B), unlock(B), unlock(A), commit。判断是否满足 2PL 与严格 2PL。

评分点和解析
3 分第一次释放是 unlock(B)，之后没有再申请新锁，所以满足 2PL。
3 分写锁 A 在 commit 前被 unlock(A) 释放，若严格 2PL 要求写锁保持到 commit/abort，则不满足严格 2PL。

关键区别：2PL 看“释放后是否又加锁”；严格 2PL 看“写锁是否持有到事务结束”。

题 5：恢复分类

06:00

日志顺序：<T1 start>, <T1, A, 10, 20>, <T2 start>, <T2, B, 5, 8>, <T1 commit>, checkpoint, <T3 start>, <T3, C, 7, 9>, crash。崩溃恢复时 Undo/Redo 怎么分？

评分点和解析
3 分T1 已 commit，属于 Redo 或确认集合；T2 没有 commit，若检查点时仍未提交且修改可能存在，需要 Undo；T3 start 后未 commit，需要 Undo。
2 分Undo 使用旧值：T2 把 B 恢复为 5，T3 把 C 恢复为 7。
2 分Redo 使用新值：T1 把 A 确认为 20。若题目说明 checkpoint 已把 T1 持久化，可写 Redo 或确认无需操作，但必须说明原因。

考试稳妥写法：先列集合，再谈值。集合写错，后面数值通常全崩。

题 6：ER 图到关系模式

07:00

图书馆系统：读者可借阅多本图书，每本图书可被不同读者多次借阅。每次借阅记录借阅日期、归还日期。请给出实体、联系、基数，并写出关系模式起稿。

评分点和解析
2 分实体：读者、图书。借阅可作为联系，也可实体化为借阅记录，因为它有借阅日期、归还日期等属性。
2 分基数：读者与图书是 M:N；一次借阅记录对应一个读者和一本图书。
3 分关系模式：读者(读者号, 姓名, ...)，图书(图书号, 书名, ...)，借阅(读者号, 图书号, 借阅日期, 归还日期, ...)。若同一读者可多次借同一本书，借阅主键应包含借阅日期或单独借阅编号。

高分提醒：只写“读者-图书 M:N”还不够。联系有自己的属性时，必须把联系转成关系表。

5. 闭眼复述：这才是真的会

下面这 5 句话，如果你能不看页面说出来，数据库 Day3 后半就基本进入可调用状态。

候选键：右部缺谁
BCNF：左边超码
调度：冲突三条件
恢复：提交状态
复述标准答案

候选键：先找不在依赖右部的属性，它们通常必须保留；再补属性求闭包，能推出全部是超码，删任一属性不行才是候选键。

BCNF：每条非平凡依赖 X->Y 都要求 X 是超码；若违反，用 R1=X∪Y、R2=R-(Y-X) 分解，并继续检查。

冲突图：只有不同事务、同一数据项、至少一个写才冲突；先出现的事务指向后出现的事务；图无环才冲突可串行化。

2PL：第一次 unlock 后不能再申请新锁；严格 2PL 通常要求写锁持有到 commit/abort。

恢复：崩溃时未提交事务 Undo，用旧值；已提交事务 Redo 或确认，用新值。

ER：实体是要记录的对象，联系是对象之间的关系，联系有属性或 M:N 时通常转成独立关系表。
