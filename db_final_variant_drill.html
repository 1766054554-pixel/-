1. 先把题分进正确盒子

数据库题最怕的不是难，而是题型没分清就动笔。比如“全部学生都选修”不是普通 JOIN，“成绩超过 10 人才统计”不是 WHERE，“释放锁后又申请锁”不是恢复题，而是 2PL 题。下面这张地图是你做题前的脑内分诊台。

SQL 查询题干出现“检索、统计、平均、至少、全部、没有、最高”。第一步：圈表、连接条件、分组字段、筛选发生在行还是组。
关系代数题干明说“用关系代数表达”。第一步：先写输出列 Π，再找筛行 σ、连接 ⋈、差集 −、除法 ÷。
DML / 视图题干出现插入、删除、修改、视图、WITH CHECK OPTION。第一步：先确定目标表，再用 SELECT 子查询找被操作的行。
事务并发题干出现 read/write、lock/unlock、S/X/U、schedule。第一步：冲突图或第一次 unlock，不要先谈 Undo/Redo。
恢复日志题干出现 crash、START、COMMIT、old/new value、Undo/Redo。第一步：分 winner/loser，再决定重做还是撤销。
范式与设计题干出现 R(A,B,C)、F={...}、BCNF、无损、ER、业务需求。第一步：范式求闭包；ER 圈实体/属性/联系/基数。
你现在的做法：先做第 2 节 20 题，不会就看解析；再挑第 3 节大题骨架写 3 行：题眼、第一步、最怕错点。最后再去做 2023A 整卷，效果会比硬刷整卷好很多。

3. 8 个大题骨架：从题眼写到可交卷答案

这部分不是让你背八道题，而是练八种“考试会换皮”的答题模板。每道先自己写，再展开答案。写不完整也没关系，但一定要写出第一步。

大题 1｜SQL 全称查询：选修了全部课程的学生

关系：S(S#, SNAME, AGE, SEX)，SC(S#, C#, GRADE)，C(C#, CNAME, TEACHER)。写 SQL：检索选修了全部课程的学生学号和姓名。

展开标准答案和换问法解析

计数法：

SELECT s.S#, s.SNAME
FROM S s
JOIN SC sc ON s.S# = sc.S#
GROUP BY s.S#, s.SNAME
HAVING COUNT(DISTINCT sc.C#) = (SELECT COUNT(*) FROM C);

双重 NOT EXISTS：

SELECT s.S#, s.SNAME
FROM S s
WHERE NOT EXISTS (
  SELECT *
  FROM C c
  WHERE NOT EXISTS (
    SELECT *
    FROM SC sc
    WHERE sc.S# = s.S# AND sc.C# = c.C#
  )
);
为什么这样“全部”是全称量词。人话是：不存在一门课程 c，使得这个学生没有选 c。计数法更短，双重 NOT EXISTS 更接近逻辑。
变式识别“选修了所有刘老师课程”“全部学生都选修的课程”“参加所有剧目”都不是普通 JOIN。看到“所有/全部/每一个”，先想全称查询。

大题 2｜SQL “没有/不学”：李想不学的课程

写 SQL：检索李想同学不学的课程号和课程名。

展开标准答案和扣分点 
SELECT c.C#, c.CNAME
FROM C c
WHERE NOT EXISTS (
  SELECT *
  FROM S s
  JOIN SC sc ON s.S# = sc.S#
  WHERE s.SNAME = '李想'
    AND sc.C# = c.C#
);
为什么这样这题从课程表 C 出发，因为答案要包含“没有被李想选”的课程。若从 SC 出发，天然只能看到已经有人选的课程，容易漏掉没人选过的课程。
变式识别“从未参加”“没有选修”“不属于”“未出现”都先从全集表出发，再排除已经发生的记录。

大题 3｜SQL 分组筛组：选课人数超过阈值

写 SQL：统计每门课程的选修人数，只显示人数超过 10 的课程，按人数降序、课程号升序。

展开标准答案和 WHERE/HAVING 区分 
SELECT sc.C#, COUNT(sc.S#) AS student_count
FROM SC sc
GROUP BY sc.C#
HAVING COUNT(sc.S#) > 10
ORDER BY student_count DESC, sc.C# ASC;
为什么这样WHERE 在分组前筛“某一行”，HAVING 在分组后筛“某一组”。“人数超过 10”必须等每门课人数算出来后才能判断，所以用 HAVING。
变式识别“平均成绩超过 80 的课程”“选课人数不少于 2 的学生”“重修人数最多的课程”都先 GROUP BY，再 HAVING/ALL/回表。

大题 4｜SQL DML：插入、修改、删除都先找目标表

写三个语句骨架：1. 把所有女生的物理课选课记录插入 SC；2. 把物理课成绩加 20 但不超过 100；3. 删除选修 LIU 老师课程的女同学选课元组。

展开标准答案和 DML 陷阱 
-- 1. 插入女生的物理课记录，假设物理课课程号为 C001
INSERT INTO SC(C#, S#)
SELECT 'C001', s.S#
FROM S s
WHERE s.SEX = 'F';

-- 2. 成绩加 20 但不超过 100
UPDATE SC
SET GRADE = CASE
  WHEN GRADE + 20 > 100 THEN 100
  ELSE GRADE + 20
END
WHERE C# = 'C001';

-- 3. 删除选修 LIU 老师课程的女同学选课元组
DELETE FROM SC
WHERE S# IN (SELECT S# FROM S WHERE SEX = 'F')
  AND C# IN (SELECT C# FROM C WHERE TEACHER = 'LIU');
为什么这样DML 题先问“要改哪张表”。第 3 题删的是女同学的选课元组，所以目标表是 SC；如果写 DELETE FROM S 就变成删除学生本人，题意变了。
变式识别“插入一批满足条件的记录”是 INSERT INTO SELECT；“分段修改”用 CASE；“删除某类关系记录”通常 DELETE 关联表，不一定删实体表。

大题 5｜关系代数：参加了全部剧目的演员

Actor(actor_id, name, year_born)，Play(play_id, title, author, year_written)，Role(actor_id, character_name, play_id)。用关系代数写出：参加过所有剧目的演员姓名。

展开关系代数答案和符号读法 
AllActor = Π_actor_id,play_id(Role) ÷ Π_play_id(Play)
Result = Π_name(Actor ⋈ AllActor)
为什么这样除法的结果保留“对所有 play_id 都配对出现的 actor_id”。先用 Role 拿到演员-剧目二元关系，再除以 Play 的全集 play_id，最后连接 Actor 取姓名。
变式识别“所有课程/全部剧目/每个商品/全部学生”在关系代数里优先想到 ÷。如果只是“参加过某个剧目”，才是普通 σ + ⋈ + Π。

大题 6｜事务并发：冲突图与 2PL 不是同一问

调度：r1(A), w1(A), r2(A), r2(B), r1(B), w1(B)。判断是否冲突可串行化，并说明如何用 2PL 让同一事务集合生成可串行化调度。

展开答案骨架和常见错法

冲突图：A 上有 w1(A) 在 r2(A) 前，得边 T1 -> T2。B 上有 r2(B) 在 w1(B) 前，得边 T2 -> T1。图有环，所以该调度不是冲突可串行化。

2PL 改造思路：让每个事务先获得自己需要的锁，等所有读写完成后再释放。例如 T1 先拿 A、B 的所需锁并完成 r1(A), w1(A), r1(B), w1(B)，再释放；T2 之后拿 A、B 的锁并完成读写。这样生成的调度等价于串行顺序 T1, T2。

为什么这样冲突可串行化看的是“同一数据项、不同事务、至少一个写”。2PL 看的是“释放第一个锁之后有没有再申请新锁”。两问的证据不同，不能混答。
变式识别题干只有 r/w 就画优先图；题干有 S/X/U、lock/unlock 就查 2PL；题干有 strict 就看写锁是否持有到 commit/abort。

大题 7｜恢复日志：先分 winner / loser

日志：<START T>，<T,A,10,11>，<START U>，<U,B,20,21>，<T,C,30,31>，<U,D,40,41>，<COMMIT U>，随后崩溃。分别说 Undo logging 和 Redo logging 的恢复动作。

展开恢复答案和分流口诀

第一步：U 有 COMMIT，是 winner；T 没有 COMMIT，是 loser。

Undo logging：重点是撤销未提交事务。恢复时把 T 修改过的 A、C 恢复为旧值，即 A=10，C=30。U 已提交，不撤销。

Redo logging：重点是保证已提交事务结果写回。恢复时重做 U 修改过的 B、D，使 B=21，D=41。T 未提交，不重做。

为什么这样恢复题先不要盯每个数值乱算。先分事务状态：已提交是 winner，未提交是 loser。Undo 看 loser 旧值，Redo 看 winner 新值。
变式识别如果题干问 force/steal 或 Undo/Redo/Undo-Redo logging，要写清假设；如果问“哪些值可能/必须写盘”，要区分内存、磁盘和日志。

大题 8｜范式与 ER：一个算法，一个翻译

范式：R(A,B,C,D,E,F)，F={A -> BC, D -> AF}。求候选键，判断 BCNF 并给出一种分解。ER：客户有多个电话，账户可属于多个客户，客户最多一个联系人，如何转关系模式？

展开范式和 ER 答案

范式第一步：右部出现 B,C,A,F，没出现的是 D,E，所以每个候选键必须包含 D,E。(DE)+：D -> AF，A -> BC，所以推出 A,B,C,D,E,F，DE 是候选键。

BCNF 判断：A -> BC 中 A 不是超键，违反 BCNF；D -> AF 中 D 也不是超键，违反 BCNF。一种分解：

R1(D,A,F)
R21(D,B,C)
R22(D,E)

ER 转关系：客户 Customer(ID, name, occupation, contact_emp_id)；客户电话 CustomerPhone(customer_id, phone)；账户 Account(account_no, branch_code, balance, type)；客户-账户 AccountOwner(customer_id, account_no, branch_code)；支行 Branch(branch_code, location, employee_count)；员工 Employee(emp_id, name, salary, branch_code)。

为什么这样范式题按闭包算法，不靠感觉。ER 题按翻译规则：多值属性单独表，M:N 联系单独表，1:N 在 N 端放外键，0/1 联系可以放可空外键。
变式识别诊所、银行、演员剧目、学生选课都一样：名词找实体，动词找联系，数量词找基数，带属性联系或 M:N 单独成表。
