0. 先把题干拆成三张图各自要用的材料

题干核心如下：BUAA 信息化办公室准备开发实验上机安排系统。教师可以提交课程实验项目、安排实验机房、查询实验安排结果、打印实验安排报表。学生可以查询课程实验项目、查询课程实验安排。管理员可以管理教师、学生、课程实验等基本信息，管理新闻公告，查询实验机房时间安排。所有用户可以修改密码、查看新闻公告。未登录用户通过登录账号密码成为对应类型用户。

用例图材料 参与者：教师、学生、管理员、未登录用户。动作：提交、安排、查询、打印、管理、修改密码、查看公告、登录。
类图材料 名词：用户、教师、学生、管理员、课程实验项目、实验机房、实验安排、新闻公告、报表。
顺序图材料 具体流程：学生查询课程实验安排；管理员管理新闻。每个顺序图只画一个场景，不要塞完整系统。
画图题第一分钟：不要打开画图工具就画。先在草稿纸上写三行：角色、名词对象、具体流程。这样三张图不会互相串味。
同一个词在用例图里在类图里在顺序图里
学生参与者 ActorStudent 类或 User 子类生命线 actor 或对象
查询课程实验安排用例椭圆可能是 ScheduleService 的方法一串消息调用
新闻公告查看公告/管理公告用例NewsAnnouncement 类管理员管理新闻流程中的实体对象
三图粒度千万别混：用例图站在系统外看“谁要达成什么目标”；类图站在系统内部看“长期保存和协作的对象是什么”；顺序图站在一次操作里看“消息按什么时间顺序传”。同一个“查询安排”，在用例图是一个目标，在顺序图是一串消息，在类图通常不是类名。

1. 用例图：先问“谁使用系统做什么”

用例图不是流程图，也不是类图。它只回答系统边界和功能边界：哪些外部角色可以使用系统提供的哪些功能。

你先试：列参与者和用例

在下面写出参与者、每个参与者对应的功能，以及你会不会使用泛化/include/extend。

看参考图和拆解

参考画法：下面不是唯一标准图，但它满足考试核心：系统边界清楚、参与者清楚、主要功能覆盖、公共功能抽出来。

参与者：教师、学生、管理员、未登录用户都在系统外，所以画成 Actor。不要把“实验机房”“新闻公告”画成参与者，它们是系统管理的对象。

用例：动词短语通常变用例：提交、安排、查询、打印、管理、修改密码、查看公告、登录。

公共用例：所有登录用户都能修改密码、查看公告，可以让教师/学生/管理员都关联它们。未登录用户至少可以查看公告和登录。

关系取舍：这道题没有明确说某功能必然包含另一个功能，保守画关联即可。不要强行乱用 include/extend。

关系从零解释什么时候用这道题的建议
普通关联 参与者可以使用某个用例。 题干只说“教师可以提交、查询、打印”。 优先使用，最稳。
include A 必然包含 B，B 是 A 的公共子流程。 多个用例都必须执行“登录校验”“权限校验”这类共同步骤，且题目要求画用例关系。 题干没有明确“必含”，不强行画。
extend B 只在某些条件下插入 A，是可选/异常扩展。 题干写“如果冲突则重新安排”“可选打印报表”等条件流程。 2020 题干没有足够清楚的扩展点，保守不画。
泛化 多个参与者或用例共享更一般的父类。 教师、学生、管理员都可以抽象为“登录用户”。 可画，也可用公共用例关联；不要因此漏掉具体参与者。
用例图扣分点：漏系统边界；把业务对象画成参与者；把步骤流程画成一串椭圆；随便乱画 include/extend；未登录用户和公共功能漏掉。

2. 类图：先问“系统里有哪些长期存在的业务对象”

类图不是把所有动词都画成类。类通常来自稳定名词：用户、教师、学生、管理员、课程实验项目、机房、实验安排、新闻公告、报表。动词更可能变成方法或服务。

你先试：从题干抽候选类、属性、关系

看参考类图和讲解

参考类图：考试要求“必要的属性和方法、关系标注、数量关系”，参数类型不考，所以重点是类和关系。

继承：教师、学生、管理员都是用户类型，可以继承 User。这样公共操作如登录、修改密码不必重复写。

核心业务类：CourseExperiment、LabRoom、ExperimentSchedule、NewsAnnouncement。它们是系统长期保存和管理的对象。

关系：一个教师可提交多个课程实验项目；一个课程实验项目可产生多个安排；一个安排使用一个机房；学生查询安排但不一定拥有安排；管理员管理公告和基础信息。

多重性：类图大题常扣这里。关系两端写 1、0..*、*，至少要体现“一对多”和“多对一”。

类图关系人话判断本题怎么用不要误用成
泛化/继承 “A 是一种 B”。子类继承父类公共属性/操作。 Teacher、Student、Admin 是 User 的不同用户类型。 不要把 CourseExperiment 继承 User，它不是用户。
关联 两个类长期知道彼此或有业务关系。 Teacher 提交 CourseExperiment；Schedule 使用 LabRoom。 不要所有关系都画继承。
聚合/组合 整体-部分。组合更强，部分生命周期依赖整体。 若你把 ExperimentSchedule 看成由若干安排明细组成，可用组合；本题不强求。 不要为了图复杂强行聚合所有类。
依赖 短暂使用，不一定长期持有。 报表生成服务依赖安排数据可以用依赖；2020 题类图可省略服务类。 不要把核心业务对象只画成依赖。
多重性 一端对应另一端几个对象。 一个教师 0..* 个实验项目；一个机房 0..* 个安排；一个安排通常对应 1 个机房。 不要漏写，漏了会显得关系没说明清楚。
类图扣分点：把“查询结果”“打印报表”全都单独画类；不标多重性；类里只有名字没有属性/方法；把所有关系都画成继承；没有说明为什么抽这些类。

3. 顺序图：一张图只画一个具体场景

题目要求两个顺序图：学生查询课程实验安排、管理员管理新闻。顺序图回答“消息按时间怎么发生”。从上到下是时间，不是功能菜单。

生命线类型从零解释本题例子为什么有用
Actor 系统外部发起交互的人或角色。 学生、管理员。 告诉阅卷人这个场景由谁触发。
边界对象 界面、页面、API 入口，负责接收用户动作。 查询界面、新闻管理界面。 避免只有“用户 -> 系统”过粗。
控制对象 处理业务规则和流程的服务/控制器。 ScheduleService、NewsService。 体现“谁负责校验、组织查询、创建对象”。
实体/数据对象 长期保存的业务数据或仓储。 ScheduleRepository、NewsAnnouncement。 和类图相连，说明顺序图不是乱发消息。

顺序图 A：学生查询课程实验安排

看参考顺序图和解析

生命线：学生、查询界面、ScheduleService、ScheduleRepository。界面是边界对象，服务是控制对象，Repository/数据对象负责取数据。

消息顺序：学生发起查询，界面把请求交给服务，服务查数据，数据返回，服务组织结果，界面显示给学生。

为什么不只画“学生 -> 系统”：上机题通常允许一定设计展开。只有学生和系统两个生命线太粗，难体现顺序图能力。可以用边界/控制/实体三层。

顺序图 B：管理员管理新闻

看参考顺序图和解析

场景选择：“管理新闻”太大，顺序图要选一个具体动作。这里选择“新增新闻”。删除、修改也可以，但要在图旁说明。

生命线：管理员、新闻管理界面、NewsService、NewsRepository、NewsAnnouncement。注意 NewsAnnouncement 是实体对象，不是参与者。

消息：管理员提交新闻，界面调用服务，服务创建公告对象并保存，最后返回成功。

顺序图扣分点：一张顺序图塞所有功能；消息没有时间顺序；生命线只有参与者没有系统对象；返回箭头乱画；把类图关系画进顺序图。

4. 考场自查清单：画完必须看这几项

用例图有系统边界吗？参与者都在边界外吗？主要功能覆盖了吗？有没有乱用 include/extend？
类图候选类来自稳定名词吗？属性/方法是否必要？关系两端有多重性吗？继承/关联/聚合有没有乱用？
顺序图是否只描述一个场景？生命线是否有边界/控制/实体层次？消息是否从上到下？返回是否合理？
文字说明2020 题明确要求必要说明。每张图旁至少写一句“我为什么这样抽参与者/类/生命线”。
阅卷视角你要显式表现出来低分信号
是否读懂题干教师、学生、管理员、未登录用户和公共功能不漏。只画教师/学生，漏管理员或未登录用户。
是否懂图的语义用例图画功能边界，类图画静态结构，顺序图画一次交互。把“查询安排”同时当用例、类、状态乱放。
是否有 UML 基本规范类图有类名/属性/方法/关系/多重性；顺序图有生命线和消息顺序。只有几个框和线，没有角色、箭头、数量、说明。
是否能解释设计图旁写 1-2 句抽取理由和取舍。画得很满但无法说明为什么。
