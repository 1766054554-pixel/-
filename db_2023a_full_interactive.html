二、问答题（10 分）

1. 简述数据库设计（新奥尔良法）的基本步骤。

展开参考答案和得分点

标准骨架：需求分析 -> 概念结构设计 -> 逻辑结构设计 -> 物理结构设计 -> 数据库实施 -> 运行和维护。

  • 需求分析：搞清楚系统要管理什么数据、谁使用、有哪些业务规则。这里会产出数据字典、数据流图、需求说明。
  • 概念结构设计：把现实世界抽象成 ER 模型，找实体、属性、联系、基数。
  • 逻辑结构设计：把 ER 图转成关系模式，确定主键、外键、完整性约束，并做必要规范化。
  • 物理结构设计：考虑索引、存储、访问路径、性能，不再是“业务长什么样”，而是“数据库怎么放更快”。
  • 实施/运行维护：建库、装载数据、测试、备份、调优、权限管理。

考试提醒：如果题目问“需求分析阶段工作”，不要把数据抽象、物理存储混进去；它们分别更偏概念设计和物理设计。

2. 简述视图的定义和主要作用。

展开参考答案和得分点

视图是由查询语句定义的虚表。它本身通常不独立存储完整数据，而是在使用时根据基表查询得到结果。

  • 简化查询：把复杂连接、筛选、聚合封装成一个“看起来像表”的对象。
  • 安全控制：只暴露允许用户看的列或行，例如不让普通用户看到 salary。
  • 逻辑数据独立性：基表结构变化时，可以用视图保持外部接口相对稳定。
  • 个性化外模式：不同角色看到不同视图。

和本卷 SQL 第 6 小题的联系：WITH CHECK OPTION 用来限制通过视图更新后的数据仍满足视图条件。

三、SQL 题与关系代数（20 分）

题目背景：表 emp(num, name, dept, salary, boss)，boss 是指向 emp(num) 的外键。总经理的 boss 指向自己，其它人的 boss 指向别人。

create table emp (
  num integer not null,
  name varchar(20) not null,
  dept varchar(20) not null,
  salary integer not null,
  boss integer not null,
  primary key (num),
  foreign key (boss) references emp(num)
);

3.1 关系代数：先认清卷面符号

σ条件(R)：选择，筛行，对应 SQL 的 WHERE。读作“在 R 中筛出满足条件的行”。

Π列名(R)：投影，筛列，对应 SQL 的 SELECT。有些资料也写成小写 π，但本页统一用卷面更醒目的大写 Π，防止你考试时认不出来。

R × S：笛卡尔积，把两张表暴力两两组合。

R ⋈条件 S：连接，等价于先 R × S 再 σ。

R ∪ S：并集，两边关系结构要兼容。

R - S：差集，在 R 里但不在 S 里。

R ÷ S：除法，专门表达“全部/所有/每一个”。比如“选修了全部课程的学生”，先找 (SID, CID)，再除以全部 CID。

ρ新名(R)：重命名，自连接时给同一张表起别名。

卷面翻译顺序：先 FROM/连接，再 WHERE/选择，最后 SELECT/投影。

1. 用关系代数查找不是其他人的 boss 的员工，列出编号、姓名、部门。

展开参考答案和解析

第一动作：“不是其他人的 boss” = 所有员工 减去 当过别人 boss 的员工。

设 E = ρE(emp) 表示员工，B = ρB(emp) 表示可能的 boss。

当过别人 boss 的人：

ΠB.num,B.name,B.dept(σE.boss = B.num ∧ E.num ≠ B.num(E × B))

所以答案可写为：

Πnum,name,dept(emp) - ΠB.num,B.name,B.dept(σE.boss = B.num ∧ E.num ≠ B.num(ρE(emp) × ρB(emp)))

为什么要有 E.num ≠ B.num：总经理的 boss 指向自己，但题目说“不是其他人的 boss”，自己指向自己不算“其他人”。

2. 用关系代数查找张三的直属领导和直属下级的员工编号。

展开参考答案和解析

第一动作：把同一张 emp 拆成三种角色：张三 Z、领导 B、下级 S。

直属领导编号：

ΠB.num(σZ.name='张三' ∧ Z.boss=B.num(ρZ(emp) × ρB(emp)))

直属下级编号：

ΠS.num(σZ.name='张三' ∧ S.boss=Z.num ∧ S.num≠Z.num(ρZ(emp) × ρS(emp)))

考试可接受写法：如果题目没有要求合并成一个关系，分成“直属领导”“直属下级”两条表达式最清楚。如果要合并，记得两边列结构要一致。

3.2 SQL 六问

1. 查找姓张的员工的薪水平均值。

展开参考答案 
SELECT AVG(salary) AS avg_salary
FROM emp
WHERE name LIKE '张%';

易错点：不要写 name = '张%'。等号是精确匹配，LIKE 才能配合通配符。

2. 查找各部门薪水最高的员工编号、姓名和部门号。

展开参考答案 
SELECT e.num, e.name, e.dept
FROM emp e
JOIN (
  SELECT dept, MAX(salary) AS max_salary
  FROM emp
  GROUP BY dept
) m
ON e.dept = m.dept AND e.salary = m.max_salary;

为什么不能只写 GROUP BY dept 后 SELECT num,name：分组后一个部门可能有多人并列最高，非分组列不能随便拿；正确做法是“聚合结果回表”。

3. 查找除总经理之外每个员工及其 boss 的编号和姓名，先按 boss 姓名，再按员工姓名排序。

展开参考答案 
SELECT e.num AS emp_num, e.name AS emp_name,
       b.num AS boss_num, b.name AS boss_name
FROM emp e
JOIN emp b ON e.boss = b.num
WHERE e.num <> e.boss
ORDER BY b.name, e.name;

卷面信号：同一张表既表示员工又表示上级，就是自连接，必须起别名。

4. 查找员工平均薪资最高的部门，可能有多个。

展开参考答案 
SELECT dept
FROM emp
GROUP BY dept
HAVING AVG(salary) >= ALL (
  SELECT AVG(salary)
  FROM emp
  GROUP BY dept
);

也可以把部门平均薪资先做成子查询，再找最大值。关键是保留并列最高，所以不要只取一行。

5. 查找归总经理直接领导的员工的最低薪水。

展开参考答案 
SELECT MIN(e.salary) AS min_salary
FROM emp e
JOIN emp g ON e.boss = g.num
WHERE g.num = g.boss
  AND e.num <> g.num;

为什么加 e.num <> g.num：总经理自己也满足 boss 指向总经理，如果不排除，可能把总经理自己的工资算进去。

6. 创建视图：所有薪水小于 5000 的员工，且不能通过视图把薪水调到 5000 以上。

展开参考答案 
CREATE VIEW low_salary_emp AS
SELECT num, name, salary
FROM emp
WHERE salary < 5000
WITH CHECK OPTION;

WITH CHECK OPTION 的作用：通过视图插入或更新后的记录仍必须满足 salary < 5000。这就是“视图不只会看，还会管更新边界”。

四、事务管理与恢复（20 分）

T1: r(B), w(B: 11 -> 12)
T2: r(B), w(B: 12 -> 13), r(A), w(A: 30 -> 31), COMMIT
T3: r(A), w(A: 29 -> 30), COMMIT
T1: r(D), w(D: 44 -> 45) *** , COMMIT

实际顺序是：T1 对 B，T2 对 B，T3 对 A，T2 对 A 并提交，T1 对 D，星号处故障，后面 T3/T1 尚未来得及提交。

1. 这个调度等价于什么串行调度？

展开参考答案

第一动作：画优先图。只看不同事务对同一数据项的冲突。

  • 在 B 上，T1 先写 B，T2 后读/写 B，所以有边 T1 -> T2。
  • 在 A 上，T3 先写 A，T2 后读/写 A，所以有边 T3 -> T2。
  • T1 和 T3 没有共同数据项冲突。

图无环，所以冲突可串行化。等价串行顺序中 T2 必须在 T1、T3 后面，因此可写为 T1, T3, T2 或 T3, T1, T2。

2. 这些事务遵守两阶段锁协议吗？请证明。

展开参考答案

2PL 判断口诀：只要“释放过锁之后又申请新锁”，就违反两阶段锁协议。

题目给了两种策略：

  • 策略 1：事务一开始先获得本事务所有操作需要的锁，随后每个操作完成后释放对应锁。这种策略下，锁都在开头拿完，后面只释放，不再申请新锁，所以每个事务自身满足 2PL。
  • 策略 2：每次读/写前才申请对应锁，操作完成后、提交前释放。若一个事务先释放了某个数据项的锁，后面又为了读/写另一个数据项申请新锁，就违反 2PL。T1 做完 B 后如果释放 B，后面又申请 D；T2 做完 B 后如果释放 B，后面又申请 A；它们都会违反。T3 只操作 A，不存在释放后再申请新锁的问题。

考试写法：先给定义，再指出具体“释放后再加锁”的位置，别只写结论。

3. 若不符合 2PL，去除最少操作使其符合 2PL；分别按两种策略分析。

展开参考答案

策略 1：每个事务开始时一次性拿到所需锁，因此不需要删除操作即可满足 2PL。

策略 2：T1 的 B 操作之后又去操作 D，T2 的 B 操作之后又去操作 A，都会形成“释放后再申请”。为了最少删除，可以删除导致第二次申请新锁的后续数据项操作，例如：

  • 删除 T1 对 D 的读写操作，避免 T1 后续再申请 D 锁。
  • 删除 T2 对 A 的读写操作，避免 T2 后续再申请 A 锁。

注意：这类题重点不是机械给唯一答案，而是说明“删掉哪些操作后，每个事务的加锁都集中在释放之前”。

4. 改造后是否为严格两阶段锁协议？为什么？

展开参考答案

严格 2PL：不仅要满足 2PL，还要求至少所有排它锁/写锁一直持有到事务提交或中止。

如果题目策略是“操作完成后立即释放该操作的锁”或“提交前释放锁”，通常就不是严格 2PL，因为写过 B/A/D 的锁可能在提交前被释放，其他事务可能读到未提交数据，导致级联回滚风险。

答题句式：该调度可以通过删操作满足普通 2PL，但不是严格 2PL；原因是写锁没有保持到 commit/abort，不能阻止其它事务读取未提交结果。

5. 星号处故障，Undo 日志恢复：回滚谁？是否级联？哪些结果能确定持久化？

展开参考答案

第一层：谁提交了？星号处 T2 已经 commit，T1 和 T3 尚未 commit。

  • 需要回滚：T1、T3。
  • 不回滚：T2，原则上已提交事务不能随便撤销。

第二层：是否有级联风险？T2 读了 T3 写出的 A=30，并把 A 写成 31 后提交；而 T3 在星号处还没提交。这说明原调度存在脏读/不可恢复风险。若系统真的允许这样执行，就会出现“已提交 T2 依赖未提交 T3”的麻烦；正规系统会用严格 2PL 或恢复协议避免它。

第三层：哪些结果能确定持久化？在纯 Undo 日志语境中，如果采用 force-at-commit，T2 的提交结果应已持久化；未提交的 T1/T3 修改不能视为最终有效，恢复时要撤销。若题目要求讨论 steal/force，应写清假设：

  • force：提交时强制写盘，T2 的提交结果可以确定持久保存。
  • no-force：提交不一定立即写盘，仅凭题干不能确定 T2 修改是否已经落盘。
  • steal：未提交事务修改可能已经写盘，所以 Undo 日志需要能撤销 T1/T3。
  • no-steal：未提交修改不会写盘，未提交结果可确定没有持久保存。

这题拿分点：不要只写“回滚 T1/T3”。必须把 T2 读 T3 未提交 A 的依赖关系指出来，说明恢复和可恢复调度之间的关系。

五、规范化理论（20 分）

关系模式 R(A,B,C,D,E)，函数依赖：

ABC -> D，E -> B，AD -> C

1. 求 R 的候选键。

展开参考答案

第一动作：看右部出现过哪些属性。右部有 B、C、D；没有出现在右部的是 A,E，所以任何候选键都必须包含 A 和 E。

求闭包：

  • (A,E,C)+：E -> B，所以有 A,E,C,B；再由 ABC -> D 得到 D。覆盖 A,B,C,D,E，所以 AEC 是键。
  • (A,E,D)+：E -> B，所以有 A,E,D,B；再由 AD -> C 得到 C。覆盖 A,B,C,D,E，所以 AED 是键。

候选键：AEC 和 AED。

为什么 AE 不是键：AE 只能推出 B，推不出 C/D；还差一个 C 或 D 来启动后续依赖。

2. 将 R 分解为 BCNF。

展开参考答案

BCNF 分解算法：找到违反 BCNF 的依赖 X -> Y，分成 R1 = X ∪ Y 和 R2 = R - (Y - X)，继续检查。

一种可行分解：

  1. 依赖 E -> B 中 E 不是 R 的超键，违反 BCNF。分解为 R1(E,B) 和 R2(A,C,D,E)。
  2. 在 R2(A,C,D,E) 中，依赖 AD -> C 仍成立，AD 在 R2 中能推出 C，但推不出 E，因此 AD 不是 R2 超键，继续分解为 R21(A,D,C) 和 R22(A,D,E)。

最终得到：{ R1(E,B), R21(A,D,C), R22(A,D,E) }。这些子关系中保留下来的非平凡依赖左部都是超键，因此满足 BCNF。

3. 分解结果是否保持函数依赖？请证明。

展开参考答案

判断方法：把原函数依赖投影到每个子关系上，再看投影依赖的并集能不能推出原 F。

  • 在 R1(E,B) 中保留 E -> B。
  • 在 R21(A,D,C) 中保留 AD -> C，同时因为有 AD -> C，在这个子关系里 AD 是键。
  • ABC -> D 涉及 A、B、C、D，分散在多个子关系中，不能在单个子关系中检查。

用投影依赖尝试求 (ABC)+：由 E -> B 不能反推 E；由 AD -> C 需要 D 才能用。无法推出 D，所以 ABC -> D 不被保持。

结论：该 BCNF 分解无损，但不保持所有函数依赖。考试里这很常见：BCNF 和依赖保持不一定同时得到。

4. 该分解是不是无损连接？请证明。

展开参考答案

二元分解无损判据：把 R 分解成 R1、R2，若 (R1 ∩ R2) -> R1 或 (R1 ∩ R2) -> R2，则无损。

  1. 第一次分解：R -> R1(E,B) 和 R2(A,C,D,E)。交集是 E，且有 E -> B，所以 E -> EB，第一次无损。
  2. 第二次分解：R2(A,C,D,E) -> R21(A,D,C) 和 R22(A,D,E)。交集是 AD，且有 AD -> C，所以 AD -> ADC，第二次无损。

结论：连续两次无损分解，因此整体无损连接。

六、数据库设计：诊所管理系统 ER 图（10 分）

根据需求画 ER 图。你认为不确定的地方可以注释说明。

展开参考答案和画图检查清单

第一动作：先抽“名词”做实体，再看“每次/多项/多种”决定联系和基数。

实体与属性

  • 患者 Patient：患者编号 PID，姓名，年龄。
  • 医生 Doctor：姓名（题目指定为标识），年龄，科室，职称，职位。
  • 门诊记录 Visit：门诊编号 VID，门诊时间，诊断。
  • 检查项目 CheckItem：检查名称，价格，检查说明。
  • 药品 Drug：药品名称，价格。
  • 检查单/检查明细 VisitCheck：门诊编号，检查名称，检查时间。
  • 药品单/用药明细 VisitDrug：门诊编号，药品名称。

联系与基数

  • 患者 1 : N 门诊记录：一位患者可以多次就诊，每次门诊属于一位患者。
  • 医生 1 : N 门诊记录：一位医生可接诊多次，每次门诊对应一位医生。
  • 门诊记录 1 : N 检查明细，检查明细 N : 1 检查项目。
  • 门诊记录 1 : N 用药明细，用药明细 N : 1 药品。

容易漏的点：检查项目和药品不是普通字符串，因为系统要记录“每种检查/每种药品”的价格和说明，所以它们应作为实体。检查单、药品单里出现“可能有多项/多种”，通常要用明细关系表承接多值。

卷面注释可以写：若一次门诊最多只有一个检查单和一个药品单，则检查单/药品单可作为弱实体或明细集合；若题目只要求概念 ER，画 Visit 与 CheckItem/Drug 的多对多联系并给联系属性也可接受，但要说明检查时间属于检查联系。
